N-Tier Architecture

จาก Layer (Logical) สู่ Tier (Physical)

3

🏗️ Layer vs Tier ต่างกันอย่างไร?

Layer = Logical (มองไม่เห็นจับต้องไม่ได้)

การจัดระเบียบ Code ภายในโปรเจกต์เดียวกัน

  • เช่น MVC (Model-View-Controller)
  • อยู่ใน Server เครื่องเดียวกันหมด (Monolith)
  • เน้นเรื่อง Separation of Concerns (SoC)

Tier = Physical (ทางกายภาพ)

การแยก Deployment / Server ออกจากกัน

  • แยก Frontend Server, Backend Server, Database Server
  • สื่อสารกันผ่าน Network (HTTP, TCP)
  • เน้นเรื่อง Scalability & Security

Part 2

Caching Strategy

ทำไมต้อง Cache? และ Pattern ที่ได้รับความนิยม

7

⚡ ทำไมต้องมี Caching?

Database (เช่น PostgreSQL/MySQL) เก็บข้อมูลลง Harddisk (SSD) ซึ่งมีความเร็วระดับหนึ่ง แต่มักจะกลายเป็น คอขวด (Bottleneck) เมื่อมีคน Query ข้อมูลเดิมซ้ำๆ พร้อมกันหลักหมื่นคน

ทางแก้: นำข้อมูลที่ถูกถามบ่อยๆ มาเก็บไว้ใน RAM ซึ่งอ่านเร็วกว่า Harddisk เป็นพันเท่า!

💡 Analogy (เปรียบเทียบ)

เหมือนการท่องสูตรคูณ

  • ไม่ใช้ Cache: ต้องหยิบสมุดมาเปิดดูทุกครั้ง (อ่านจาก Database)
  • ใช้ Cache: จำไว้ในหัว ถามปุ๊บตอบปั๊บ (อ่านจาก RAM / Redis)
8

🚀 รู้จักกับ Redis (REmote DIctionary Server)

  • เป็น In-Memory Data Structure Store
  • เก็บข้อมูลในรูปแบบ Key-Value
  • ทำงานบน RAM ทั้งหมด (ไวมาก)
  • มักใช้ทำ Caching, Session Management, Message Broker
[แทรกรูปโลโก้ Redis หรือการทำงานของ RAM vs Disk]
9

📖 Cache-Aside Pattern (จังหวะอ่านข้อมูล)

เป็น Pattern ที่นิยมที่สุด ตัว Application จะเป็นคนควบคุมตรรกะทั้งหมด

  1. แอปพลิเคชันรับ Request
  2. ตรวจสอบ Cache ก่อน: มีข้อมูลไหม?
    • มี (Cache Hit): ส่งข้อมูลกลับให้ User เลย (จบกระบวนการ ไวมาก)
    • ไม่มี (Cache Miss): ไปต่อข้อ 3
  3. Query Database: ดึงข้อมูลจากฐานข้อมูลหลัก
  4. Save to Cache: นำข้อมูลที่ได้ไปแปะไว้ใน Cache สำหรับคนต่อไป
  5. ส่งข้อมูลกลับให้ User

⚠️ ปัญหาโลกแตกของ Cache: ข้อมูลไม่ตรงกัน!

ถ้าแก้ไขข้อมูลใน Database แต่ลืมแก้ใน Cache ผู้ใช้จะเห็นข้อมูลเก่า (Stale Data)

1. Cache Invalidation (การล้างแคช)

ทุกครั้งที่มีการ Write/Update/Delete ลง Database แอปพลิเคชันต้องสั่งลบ (Delete Key) ข้อมูลนั้นออกจาก Redis ด้วย เพื่อบังคับให้รอบถัดไปเกิด Cache Miss

2. TTL (Time-To-Live)

ตั้งเวลาหมดอายุให้ Cache อัตโนมัติ (เช่น 60 วินาที) เป็นตาข่ายรองรับความปลอดภัย หากลืม Invalidate ข้อมูลก็จะไม่อัปเดตเกิน 60 วินาที

Part 3

Load Balancing

การกระจายโหลดและ Horizontal Scaling

13

⚖️ Scaling: ทำอย่างไรเมื่อรับโหลดไม่ไหว?

Vertical Scaling (Scale Up)

อัปเกรดเครื่องเดิมให้แรงขึ้น (เพิ่ม CPU, RAM)

  • ✅ ทำง่าย ไม่ต้องแก้ Code
  • ❌ มีขีดจำกัด (Hardware ตัน)
  • ❌ แพง และเกิด Single Point of Failure

Horizontal Scaling (Scale Out)

เพิ่มจำนวนเครื่อง Server ให้ทำงานช่วยกัน

  • ✅ ขยายได้ไม่จำกัด
  • ✅ มี High Availability (เครื่องนึงพัง เครื่องอื่นทำแทน)
  • ❌ ระบบซับซ้อนขึ้น ต้องมี Load Balancer
14

🚦 Load Balancer คืออะไร?

คือ "จราจร" ที่ยืนอยู่หน้าระบบ ทำหน้าที่รับ Request จากผู้ใช้ และตัดสินใจว่าจะส่ง Request นั้นไปให้ Server ตัวไหน (App 1, App 2 หรือ App 3) เพื่อให้งานเฉลี่ยเท่าๆ กัน

🛠️ Nginx (เอ็นจินเอ็กซ์)

ใน Lab สัปดาห์นี้ เราจะใช้ Nginx ทำหน้าที่เป็นทั้ง Reverse Proxy และ Load Balancer คอยแจกจ่ายงานให้ Node.js Container ทั้ง 3 ตัว

15

🔄 Algorithm การกระจายโหลด

Load Balancer มีวิธีการเลือก Server หลายแบบ ที่นิยมใช้คือ:

Algorithm หลักการทำงาน เหมาะสำหรับ
Round-Robin (ใช้ใน Lab) วนส่งไปทีละเครื่องตามลำดับ (1 -> 2 -> 3 -> 1...) Server ทุกเครื่องมีสเปคเท่ากัน
Least Connections ส่งไปเครื่องที่กำลังรับงานน้อยที่สุด งานแต่ละ Request ใช้เวลาประมวลผลไม่เท่ากัน
IP Hash จำ IP ผู้ใช้ แล้วส่งไปเครื่องเดิมเสมอ ระบบที่ต้องการ Session Sticky (ห้ามเปลี่ยนเครื่อง)

🩺 Health Checks (ตรวจสุขภาพ Server)

ถ้า Server ตัวที่ 2 ล่ม (Crash) ไป Load Balancer จะรู้ได้อย่างไร?

คำตอบ: Load Balancer จะทำการ PING หรือส่ง HTTP Request ไปถาม Server เป็นระยะๆ เรียกว่า Health Check

  • ถ้าตอบ 200 OK -> ส่งงานให้ปกติ
  • ถ้า Time out / 500 -> ตัดออกจากระบบชั่วคราว ไม่ส่ง User ไปหา

💡 ข้อคิด (SE Concept)

นี่คือหลักการของ Fault Tolerance (ความทนทานต่อความบกพร่อง) ระบบที่ดีย่อมไม่ล่มทั้งระบบเพียงเพราะ Component เดียวพัง

Quality Attributes & Trade-offs Analysis

โลกของการออกแบบซอฟต์แวร์ ไม่มีทางออกที่สมบูรณ์แบบ มีแต่การแลกเปลี่ยน

⚖️ สิ่งที่ได้มา แลกกับ สิ่งที่เสียไป (Trade-offs)

✅ สิ่งที่ได้เพิ่ม (Benefits)

  • Performance: เร็วขึ้นด้วย Cache (ลดเวลา Query)
  • Scalability: รองรับคนได้มากขึ้นด้วย Horizontal Scaling
  • Availability: ระบบเสถียรขึ้น ล่มยากขึ้นด้วย Load Balancer & Health Check

❌ สิ่งที่เสียไป (Costs)

  • Simplicity: ระบบซับซ้อนขึ้นอย่างมาก (มีหลาย Container, config เพิ่ม)
  • Consistency: ความท้าทายเรื่อง Cache Invalidation (ข้อมูลเก่า/ใหม่)
  • Maintenance: Debug ยากขึ้น (Log กระจายอยู่หลายเครื่อง)
