📞 1.1 Synchronous Communication

การสื่อสารแบบ Request/Response

หลักการ: ผู้ส่งส่ง Request → รอ → ผู้รับประมวลผล → ส่ง Response กลับ

เปรียบเทียบในชีวิตจริง: โทรศัพท์ 📞
  • ต้องรอให้อีกฝ่ายรับสาย
  • ถ้าไม่รับ → ไม่ได้คุย (fail)
  • ต้องจบเรื่องหนึ่งก่อน ถึงจะไปทำเรื่องต่อไป

Client ต้องรอทุกขั้นตอนเสร็จ (เช่น Insert DB + Send Email + Log) ก่อนได้ 201 Created

💬 1.1 Asynchronous Communication

การสื่อสารแบบ Event-Based (ไม่ต้องรอ)

หลักการ: ผู้ส่ง Publish Event → ไม่รอ → ผู้รับ Subscribe แล้วทำงานเอง

เปรียบเทียบในชีวิตจริง: ส่ง LINE 💬
  • ส่งข้อความแล้วก็จบ ไม่ต้องรอ
  • อีกฝ่ายมาอ่านเมื่อไหร่ก็ได้
  • ปิดเครื่อง ข้อความยังอยู่ รอจนเปิดมาอ่าน

Client ได้ Response ทันทีหลัง INSERT DB เสร็จ! ที่เหลือทำต่อเป็น Background

📊 เปรียบเทียบ Sync vs Async

หัวข้อ Synchronous (Request/Response) Asynchronous (Event-Driven)
การทำงาน ส่ง → รอ → รับ ส่ง → ทำต่อเลย
Coupling Tight (ผูกแน่น) Loose (แยกจากกัน)
Failure Impact 1 ล่ม = ทั้งหมดล่ม 1 ล่ม = ที่เหลือยังทำงาน
Data Consistency Strong (ทันที) Eventual (สุดท้ายจะตรงกัน)

🤔 เมื่อไหร่ควรใช้แบบไหน?

ใชั Synchronous เมื่อ:

  • Client ต้องรู้ผลลัพธ์ทันที (เช่น โอนเงินสำเร็จหรือไม่?)
  • ต้องการ Strong Consistency (ข้อมูลต้องตรงกัน 100% ทันที)
  • ขั้นตอนถัดไปขึ้นกับผลลัพธ์ก่อนหน้า

ใช้ Asynchronous เมื่อ:

  • Client ไม่ต้องรอผลลัพธ์ (เช่น ส่ง email แจ้งเตือน)
  • ต้องกระจายงานไปหลาย Services (เช่น แจ้งคลัง + ส่ง Email)
  • ต้องการ Fault Tolerance (Service ล่ม งานรอในคิวได้)

📡 1.2 Event-Driven Architecture (EDA) คืออะไร?

รูปแบบสถาปัตยกรรมที่ Services สื่อสารกันผ่าน Events (เหตุการณ์ที่เกิดขึ้น) แทนที่จะเรียกกันโดยตรง

แนวคิดหลัก (Hollywood Principle)

"Don't call us, we'll call you"

  • แบบเก่า: Service A ต้องรู้จักและโทรหา Service B โดยตรง
  • แบบใหม่: Service A ประกาศว่ามีอะไรเกิดขึ้น ใครสนใจก็รับไป

A ไม่ต้องรู้จัก B, C, D แค่ประกาศ Event ลง Broker

🧩 3 องค์ประกอบหลักของ EDA

  1. EVENT (เหตุการณ์)

    สิ่งที่เกิดขึ้นแล้ว (Past Tense) เป็น Immutable แก้ไขไม่ได้ เช่น TaskCreated

  2. EVENT PRODUCER (ผู้ส่ง)

    Service ที่ "ประกาศ" ว่ามี Event เกิดขึ้น (ไม่ต้องรู้ว่าใครจะรับ)

  3. EVENT CONSUMER (ผู้รับ)

    Service ที่ "สมัครรับฟัง" Events ที่สนใจ (ไม่ต้องรู้ว่า Event มาจากไหน)

🔗 Loose Coupling — ข้อดีที่สำคัญที่สุด

❌ Tight Coupling (Week 6) 

async function createTask(data) {
  const task = await db.insert(data);
  await emailService.send(task);  ←
  await auditService.log(task);   ←
  return task;
}

เพิ่ม Service ใหม่ = ต้องแก้โค้ด Task API เสมอ!

✅ Loose Coupling (Week 7) 

async function createTask(data){
  const task = await db.insert(data);
  publish('TASK_CREATED', task); ←
  return task;
}

เพิ่ม Consumer ใหม่เท่าไหร่ก็ได้ Task API ไม่ต้องแก้โค้ด!

🏪 ตัวอย่างจริงในระบบที่คุ้นเคย

  • Shopee — สั่งซื้อสินค้า:
    ลูกค้ากด "สั่งซื้อ" ➔ เห็น "สั่งซื้อสำเร็จ" ทันที ➔ เบื้องหลัง OrderPlaced Event ถูกส่งไปตัดสต็อก, ตัดเงิน, แจ้งร้านค้า
  • LINE — ส่งข้อความ:
    ผู้ใช้ส่ง ➔ เห็นติ๊กถูก ✓ ทันที ➔ MessageSent Event ถูกส่งไปแจ้งเตือนปลายทาง, นับสถิติ
  • หมอพร้อม — จองวัคซีน:
    จองสำเร็จ ➔ AppointmentBooked Event ➔ ส่ง SMS ยืนยัน, เพิ่มปฏิทิน, จองโควตา

🐰 1.3 Message Broker คืออะไร?

ตัวกลาง (Middleware) ที่รับ-ส่ง-จัดการ Messages ระหว่าง Services

  • รับ Message จาก Producer
  • เก็บ Message ไว้ใน Queue (ไม่หายถ้า Service ปลายทางล่ม)
  • ส่ง Message ไปยัง Consumer ที่สมัครรับ
เหมือนระบบไปรษณีย์ 📮 คุณหย่อนจดหมายลงตู้ (Broker) เดี๋ยวไปรษณีย์จัดการส่งให้ถึงมือผู้รับเอง ไม่ต้องเดินไปส่งเอง

⚙️ RabbitMQ Architecture

ภาพรวมสถาปัตยกรรม 5 ส่วน: Producer, Exchange, Queue, Binding, Consumer

📦 องค์ประกอบทั้ง 5 ของ RabbitMQ

  • ① Producer: Service ที่ส่ง Message เข้าสู่ Exchange
  • ② Exchange: ตัวกระจาย Message ไปตามกฎ (เหมือนตู้ไปรษณีย์ส่วนกลาง)
  • ③ Queue: คิวเก็บ Message รอ Consumer มารับ (เหมือนตู้จดหมายหน้าบ้าน)
  • ④ Binding: กฎการ Route ที่เชื่อม Exchange กับ Queue
  • ⑤ Consumer: Service ที่รับ Message ไปทำงาน แล้วส่ง ACK กลับ

🔀 Exchange Types กฎการกระจายข้อความ

1. Fanout ⭐ (ใช้ใน Lab)

ส่ง Message ไปทุก Queue ที่ bind (เหมือนกระจายเสียงวิทยุ FM)

2. Direct

ส่งเฉพาะ Queue ที่มี routing key ตรงกันพอดี (ส่งจดหมายระบุชื่อ)

3. Topic

ส่งตาม pattern ของ routing key (เช่น task.*)

ทำไม Lab นี้ใช้ Fanout?
เพราะเราอยากให้ทุก Event (เช่น สร้าง Task) ส่งไปถึงทั้ง Notification และ Audit พร้อมๆ กัน

✅ Message Acknowledgement (ACK)

การรับประกันว่า Message ถูกประมวลผล ไม่สูญหาย!

  1. Queue ส่ง Message ให้ Consumer
  2. Consumer ประมวลผล Message สำเร็จ
  3. Consumer ส่ง ACK กลับ ➔ Queue ลบ Message ออก
💥 ถ้า Consumer ล่มก่อนส่ง ACK?
Queue รู้ว่างานยังไม่เสร็จ ➔ จะส่ง Message นั้นให้ Consumer ตัวอื่น หรือรอส่งให้ตัวเดิมเมื่อฟื้นคืนชีพ (Message ไม่มีวันหาย!) 🛡️

🌊 Event Flow: POST /api/tasks

เส้นประสีแดงคือ Async Boundary – Client ได้ Response ทันที ที่เหลือทำอยู่เบื้องหลัง

📋 Event Contract (สัญญาการสื่อสาร)

เพื่อไม่ให้ Consumer พัง เราต้องตกลงกันว่าโครงสร้าง (Schema) ของ Event หน้าตาเป็นอย่างไร 

{
  "id": "evt-1708600001-abc123",    // Unique ID
  "type": "TASK_CREATED",           // Event Type (สำคัญมาก!)
  "data": {                         // ข้อมูลจริงของ Event
    "id": 8, "title": "ทำ Lab Week 7", "status": "TODO"
  },
  "metadata": {                     // ข้อมูล Meta
    "timestamp": "2025-02-22T10:30:00Z",
    "source": "task-service"
  }
}

เปรียบเทียบ: เหมือนการตกลง JSON body schema ใน REST API

⏳ Eventual Consistency

ความสอดคล้องแบบ "สุดท้าย" - Trade-off ของ Async

ความหมาย

"ถ้าไม่มี Event ใหม่เข้ามา สุดท้ายทุก Service จะมีข้อมูลตรงกัน แต่อาจมีช่วงเวลาสั้นๆ ที่ข้อมูลไม่ตรงกัน"

  • ตัวอย่าง: สั่ง Shopee เห็นสำเร็จทันที แต่อีเมลมาอีก 30 วินาทีหลัง
  • ข้อระวัง: ใช้กับระบบโอนเงินไม่ได้ เพราะยอดต้องตรงทันที (Strong Consistency)

⚖️ 1.5 Quality Attributes, Trade-offs

Quality Attribute Week 6 (N-Tier) Week 7 (Event-Driven)
Response Time ~2ms (รอทุกส่วน) ~50ms (Client ไม่ต้องรอ)
Scalability / Availability ★★★★☆ ★★★★★ (Scale อิสระ, Event รอใน Queue ได้)
Consistency Strong (ทันที) Eventual (ไม่ Sync ทันที)
Complexity / Debugging ปานกลาง สูง (ต้อง Trace ข้าม Services)
