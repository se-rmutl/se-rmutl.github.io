Part 1

สร้าง Auth Service 🔑

ด่านแรกของระบบ: เช็ครหัสผ่าน ➔ ออกบัตรผ่าน (JWT)

🔑 Part 1: Auth Service (Database Schema)

Service นี้ต้องมี Database ของตัวเอง (Auth DB) เพื่อเก็บข้อมูล Credentials

⚠️ กฎเหล็ก Security

ตาราง auth_users จะเก็บ password_hash ด้วยอัลกอริทึม Bcrypt ห้ามเก็บรหัสผ่านเป็น Plaintext เด็ดขาด!

ตัวอย่าง Schema (auth-service/init.sql):

CREATE TABLE auth_users (
  id SERIAL PRIMARY KEY,
  user_id VARCHAR(50) UNIQUE NOT NULL,
  email VARCHAR(255) UNIQUE NOT NULL,
  password_hash VARCHAR(255) NOT NULL,
  role VARCHAR(20) DEFAULT 'member'
);

💻 โค้ดสำคัญ: Login & ออก JWT (auth-service)

const bcrypt = require('bcryptjs');
const jwt = require('jsonwebtoken');

app.post('/api/auth/login', async (req, res) => {
  const { email, password } = req.body;
  // 1. ดึง User จาก DB ด้วย Email (สมมติได้ตัวแปร user มา)
  
  // 2. เทียบรหัสผ่านด้วย bcrypt
  const isMatch = await bcrypt.compare(password, user.password_hash);
  if (!isMatch) return res.status(401).json({ error: 'Invalid credentials' });

  // 3. สร้าง JWT Token (Payload ห้ามใส่ Password!)
  const token = jwt.sign(
    { user_id: user.user_id, role: user.role }, 
    process.env.JWT_SECRET, 
    { expiresIn: '1h' }
  );
  
  res.json({ message: 'Login success', token });
});

Part 2

สร้าง User Service 👤

แยกการจัดการข้อมูลส่วนตัวโปรไฟล์ ออกจากรหัสผ่าน

👤 Part 2: หน้าที่ของ User Service

🌟 หน้าที่หลัก
  • เก็บข้อมูลส่วนตัวที่เปิดเผยได้ เช่น ชื่อ นามสกุล รูปโปรไฟล์ แผนก
  • ไม่ต้องเก็บ Password แล้ว (หน้าที่นั้น Auth Service เอาไปทำ)
  • เชื่อมโยงข้อมูลกับ Auth Service ด้วย user_id เดียวกัน

ตัวอย่าง Schema (user-service/init.sql):

CREATE TABLE user_profiles (
  user_id VARCHAR(50) PRIMARY KEY,
  full_name VARCHAR(100),
  department VARCHAR(50),
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

* หมายเหตุ: การดึง Profile ตัวเองก็ต้องใช้ JWT Token ยืนยันเช่นกัน!

Part 3

สร้าง Task Service พร้อม JWT Guard 🛡️

สร้างด่านตรวจไม่ให้คนนอกเข้ามาล้วงข้อมูล

🛡️ Part 3: โค้ดด่านตรวจ JWT (Middleware)

ใน Task Service เราจะเขียน authMiddleware.js ดักไว้ก่อนเข้าถึง Database

const jwt = require('jsonwebtoken');

const verifyToken = (req, res, next) => {
  // 1. มองหา Header ชื่อ Authorization
  const authHeader = req.headers['authorization'];
  if (!authHeader) return res.status(403).json({ error: 'No token provided' });

  const token = authHeader.split(' ')[1]; // ตัดคำว่า "Bearer " ออก
  
  // 2. ใช้ Secret Key ถอดรหัสเพื่อเช็คความถูกต้อง
  jwt.verify(token, process.env.JWT_SECRET, (err, decoded) => {
    if (err) return res.status(401).json({ error: 'Unauthorized!' });
    req.user = decoded; // แนบ Payload (user_id, role) ให้ Service ไปใช้ต่อ
    next(); // ให้ผ่าน! ✅
  });
};

// ตัวอย่างการใช้งาน
app.get('/api/tasks', verifyToken, (req, res) => { ... });

Part 4

API Gateway (Nginx) + Rate Limiting 🚪

ประตูทางเข้าบานเดียวของระบบ ป้องกันโจมตีแบบ DoS

🚪 Part 4: Nginx และ Rate Limiting

Reverse Proxy: ให้ Client วิ่งมาที่ Nginx (Port 80) ที่เดียว แล้ว Nginx จะทำการ Routing ไปหา Auth, User, Task Service เอง

Rate Limiting: ป้องกัน Hacker ใช้สคริปต์สุ่มรหัสผ่าน (Brute-force) หรือยิง Request จำนวนมหาศาลเพื่อทำให้ Server ล่ม (DoS Attack)

💻 โค้ด Nginx (nginx.conf)
http {
  # กำหนด Zone จำกัด 10 requests / วินาที ต่อ 1 IP
  limit_req_zone $binary_remote_addr zone=mylimit:10m rate=10r/s;

  server {
    location /api/auth/ {
      limit_req zone=mylimit burst=5 nodelay; # เปิดใช้ Rate Limit
      proxy_pass http://auth-service:3001/;
    }
    location /api/tasks/ {
      proxy_pass http://task-service:3002/;
    }
  }
}

Part 5

Centralized Logging 📊

รวบรวม Log จากทุก Microservices มาไว้ที่เดียวด้วย Loki + Grafana

📊 Part 5: ตั้งค่า Winston Logger (JSON Format)

ใน Production เราไม่ใช้แค่ console.log() เพราะมันค้นหายาก เราจะใช้ไลบรารี Winston ให้เขียน Log เป็นรูปแบบ JSON (Structured Logging)

นำโค้ดด้านขวาไปใส่ในทุก Service (Auth, User, Task)

// utils/logger.js
const winston = require('winston');

const logger = winston.createLogger({
  level: 'info',
  format: winston.format.combine(
    winston.format.timestamp(),
    winston.format.json() // Output เป็น JSON
  ),
  transports: [
    new winston.transports.Console()
  ]
});

module.exports = logger;

// การใช้งาน: logger.info('User login success', { userId: 123 });

Part 6

รวมร่างด้วย Docker Compose 🐳

จัดการ 9 Containers ให้คุยกันและตั้งค่า Security Network

🐳 Part 6: Docker Compose (Network & Port Security)

สังเกตการเปิด Port (Port Mapping) ใน docker-compose.yml

  • API Gateway (Nginx): ports: ["80:80"] เปิดให้คนภายนอกเข้าได้เพียงทางเดียว!
  • Auth, Task, User Services: ไม่มี ports ออกไปข้างนอก! ใช้แค่ expose: ["3001"] เพื่อคุยกันเองในวง LAN (Network)
  • Logging Driver: กำหนดให้ทุก Service ส่ง Log ไปหา Loki อัตโนมัติ
    logging:
      driver: loki
      options:
        loki-url: "http://localhost:3100/loki/api/v1/push"
        loki-retries: "5"

🧪 Part 7: Security Test Cases (ทดสอบระบบ)

หลังจากรัน docker compose up --build -d ให้ใช้ Postman ทดสอบ:

Test 1: เข้า API แบบไม่มี Token

ยิง GET http://localhost/api/tasks โดยไม่แนบ Header ➔ คาดหวัง: 403 Forbidden

Test 2: ขอ Token (Login)

ยิง POST http://localhost/api/auth/login พร้อม user/pass ➔ คาดหวัง: 200 OK + ได้รับ JWT Token

Test 3: นำ Token ไปดึงข้อมูล Task

ยิง GET http://localhost/api/tasks โดยแนบ Authorization: Bearer ➔ คาดหวัง: 200 OK + ได้ข้อมูล Task

🚀 ทดสอบยิง Rate Limiting (DoS Test)

ทดสอบว่า Nginx สามารถป้องกันการยิง Request สแปมรัวๆ ได้จริงหรือไม่

# รันคำสั่งนี้ใน Terminal (ส่ง Request 20 ครั้งติดๆ กันในเสี้ยววินาที)

for i in {1..20}; do curl -i http://localhost/api/auth/login; done

ผลลัพธ์ที่คาดหวัง:

  • Request แรกๆ จะทะลุไปถึง Auth Service (ได้ HTTP 200/401)
  • Request ที่เกินลิมิต 10 ครั้งต่อวินาที Nginx จะเตะทิ้งและตอบกลับด้วย HTTP 503 Service Temporarily Unavailable ทันที! (เซิร์ฟเวอร์รอดตาย)

⚖️ สรุป: Defense in Depth & Trade-offs

🛡️ การป้องกันหลายชั้น
  • ชั้นที่ 1: Rate Limiting (Nginx) กัน DoS
  • ชั้นที่ 2: JWT Validation กันคนนอก
  • ชั้นที่ 3: Role-based Auth ตรวจสิทธิ์ User
  • ชั้นที่ 4: SQL Parameterized Queries
➖ สิ่งที่ต้องแลกมา (Cost)
  • Complexity เพิ่ม: มีถึง 9 Container ดูแลยากขึ้น
  • Latency เพิ่ม: การเช็ค JWT ผ่าน Middleware กินเวลา ~5ms
  • UX: Rate Limit ตั้งน้อยไปอาจบล็อกคนใช้งานจริง

✅ Checklist งานที่ต้องส่ง (Deliverables)

  • docker compose up --build รันผ่านทุก service
  • ทดสอบ Authentication Flow สำเร็จ
  • ทดสอบ Rate Limit ได้ผลติด 503
  • เห็น Structured Log (JSON) ปรากฏใน Grafana
  • อัปเดตแผนภาพ C2 Diagram
  • เขียนเอกสาร ADR-001
