Part 2

สร้าง Auth Service 🔑

หัวใจสำคัญของการยืนยันตัวตน (Authentication)

🔑 หน้าที่ของ Auth Service

Service นี้เกิดมาเพื่อทำเรื่อง Login และเช็ค Token เท่านั้น (Single Responsibility)

2 Endpoints หลัก:

  • POST /api/auth/login
    รับ Username/Password ไปเช็คกับ Database ถ้ารหัสถูก จะ "Sign" (ออก) JWT Token
  • POST /api/auth/verify
    รับ JWT Token มาตรวจสอบลายเซ็น (Signature) ว่าของจริงไหม หมดอายุหรือยัง
⚠️ กฎเหล็กด้านความปลอดภัย

Auth Service ต้องใช้ Bcrypt เพื่อเปรียบเทียบ Password ที่แฮชไว้ใน Database ห้ามเก็บ Password เป็น Plaintext เด็ดขาด!

💻 โค้ดสำคัญ: Login & Sign Token (auth-service)

const bcrypt = require('bcryptjs');
const jwt = require('jsonwebtoken');
const SECRET_KEY = process.env.JWT_SECRET || 'supersecretkey';

app.post('/api/auth/login', async (req, res) => {
  const { username, password } = req.body;
  
  // 1. ดึงข้อมูล User จาก DB (สมมติว่าดึงมาแล้วอยู่ในตัวแปร user)
  
  // 2. เทียบรหัสผ่านด้วย bcrypt
  const isMatch = await bcrypt.compare(password, user.password);
  if (!isMatch) return res.status(401).json({ error: 'Invalid credentials' });

  // 3. สร้าง JWT Token (Payload: id, role)
  const token = jwt.sign(
    { id: user.id, role: user.role }, 
    SECRET_KEY, 
    { expiresIn: '1h' } // หมดอายุใน 1 ชั่วโมง
  );
  
  res.json({ message: 'Login successful', token });
});

Part 4

ปรับปรุง Task Service 🛡️

เปลี่ยนจาก "ใครก็เข้าได้" เป็น "ต้องมีบัตรผ่าน (JWT)"

🛡️ สร้างด่านตรวจ (JWT Middleware Guard)

ใน Task Service เราจะเขียน Middleware เพื่อดักจับทุก Request ก่อนที่จะไปยุ่งกับ Database

// task-service/server.js
const verifyToken = (req, res, next) => {
  const authHeader = req.headers['authorization'];
  if (!authHeader) return res.status(403).json({ error: 'No token provided' });

  const token = authHeader.split(' ')[1]; // ตัดคำว่า "Bearer " ออก
  
  // เรียกไปที่ Auth Service เพื่อตรวจ Token (Microservice Pattern)
  axios.post('http://auth-service:3001/api/auth/verify', { token })
    .then(response => {
      req.user = response.data.user; // เอาข้อมูล user แปะไว้ใน Request
      next(); // ให้ผ่านเข้าไปดึงข้อมูล Task ได้ ✅
    })
    .catch(err => res.status(401).json({ error: 'Unauthorized!' }));
};

// นำ Middleware ไปใช้กับ Route ที่ต้องการปกป้อง
app.get('/api/tasks', verifyToken, (req, res) => { ... });

Part 5

Nginx เป็น API Gateway 🚪

รวมศูนย์ Request และป้องกันการโจมตี (Rate Limiting)

🚪 ทำไมต้องมี API Gateway?

ในระบบ Microservices เรามีหลาย Service (Auth วิ่ง Port 3001, Task 3002) การให้ Client จำ Port ทั้งหมดเป็นเรื่องยุ่งยากและไม่ปลอดภัย

Client
เรียก Port 80
Nginx
Reverse Proxy
➜ /api/auth/ ส่งไป Auth Svc (3001)
➜ /api/users/ ส่งไป User Svc (3003)
➜ /api/tasks/ ส่งไป Task Svc (3002)

💡 นอกจากทำ Routing แล้ว Nginx ยังทำหน้าที่เป็นยามคอยกันผู้ประสงค์ร้ายได้ด้วย (เช่น บล็อก IP, จำกัด Rate Limit)

💥 Rate Limiting (ป้องกัน DoS Attack)

ปัญหา: ถ้า Hacker เขียนสคริปต์ยิง Request เข้ามา 10,000 ครั้งต่อวินาที (Spam/DoS) เซิร์ฟเวอร์เราจะพัง (Out of Memory)

วิธีแก้: ใช้ limit_req ของ Nginx เพื่อจำกัดจำนวน Request ต่อวินาที จาก IP เดียวกัน

❓ สังเกตการตั้งค่า:

limit_req_zone $binary_remote_addr zone=mylimit:10m rate=5r/s;

แปลว่า: อนุญาตให้ 1 IP ส่งคำขอได้ 5 ครั้งต่อวินาที (5r/s) ถ้าเกินกว่านี้ Nginx จะดรอปทิ้งและตอบกลับด้วย HTTP 503 (Service Unavailable) ทันที!

💻 โค้ดสำคัญ: nginx.conf

http {
    # ประกาศ Zone สำหรับ Rate Limit (5 requests per second)
    limit_req_zone $binary_remote_addr zone=apilimit:10m rate=5r/s;

    server {
        listen 80;

        # Security Headers (ป้องกัน XSS, Clickjacking)
        add_header X-Frame-Options "SAMEORIGIN";
        add_header X-XSS-Protection "1; mode=block";

        location /api/auth/ {
            limit_req zone=apilimit burst=10 nodelay; # เปิดใช้ Rate Limit
            proxy_pass http://auth-service:3001/;
        }
        # ... location อื่นๆ สำหรับ tasks, users ...
    }
}

📝 Part 6: Structured Logging ด้วย Winston

ทำไม console.log() ถึงไม่พอ?

ใน Production เรามีหลาย Container รันพร้อมกัน การดู Log ที่เป็นแค่ข้อความ (Plain Text) ทำให้ค้นหา (Search / Filter) ได้ยากมาก

Structured Logging คือการเขียน Log ออกมาในรูปแบบ JSON

❌ แบบเก่า (อ่านยาก, หาข้อมูลยาก):

User john logged in successfully at 10:00 AM

✅ แบบใหม่ Structured JSON (เอาไปเข้า ElasticSearch ได้ง่าย):

{"level":"info", "message":"User logged in", "user":"john", "timestamp":"2023-10-25T10:00:00Z"}

Part 7

ประกอบร่างด้วย Docker Compose 🐳

รันทุกอย่างขึ้นมาด้วยคำสั่งเดียว

🐳 docker-compose.yml (Network Security)

สังเกตการเปิด Port (Port Mapping) ในไฟล์ docker-compose.yml ของ Lab นี้

  • Nginx (API Gateway): ports: ["80:80"] เปิดให้คนภายนอกเข้าได้เพียงทางเดียว!
  • auth-service, task-service, mysql: ไม่มีการทำ port mapping ออกไปข้างนอก! expose: ["3001"]
🛡️ Defense in Depth (การป้องกันหลายชั้น)

นี่คือหลักการ Network Isolation แม้ Hacker จะสแกน IP ของ Server เรา ก็จะเจอแค่ Port 80 ของ Nginx ไม่สามารถยิงตรงเข้า Database (Port 3306) หรือ Microservices ภายในได้เลย!

🚀 ทดสอบยิง Rate Limiting (DoS Test)

ทดสอบว่า Nginx สามารถป้องกันการรัว Request ได้จริงหรือไม่

# รันคำสั่งนี้ใน Terminal (ส่ง Request 10 ครั้งติดๆ กันในเสี้ยววินาที)

for i in {1..10}; do curl -i http://localhost/api/auth/login; done

ผลลัพธ์ที่คาดหวัง:

  • 3-5 Request แรกจะได้ HTTP 200/401 ปกติ (ขึ้นอยู่กับ burst config)
  • Request หลังจากนั้นจะได้ HTTP 503 Service Temporarily Unavailable (Nginx บล็อกให้แล้ว!)
