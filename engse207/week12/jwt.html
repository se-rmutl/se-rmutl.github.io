Part 1

ทำไมต้องมี JWT?

รู้จักจุดอ่อนของระบบเดิม สู่กระบวนทัศน์การออกแบบระบบแบบ Stateless

🔙 ยุคก่อน JWT: Session-Based Authentication

ในอดีต (และหลายระบบในปัจจุบัน) การยืนยันตัวตนใช้ระบบ Session

1. User ส่ง Username/Password ไปที่ Server

2. Server ตรวจสอบถูกต้อง สร้าง Session ID (เช่น XYZ123) เก็บลง Database หรือ Redis

3. Server ส่ง Session ID คืนให้ User ผ่าน Cookie

4. ครั้งต่อไป User ส่ง Cookie(XYZ123) มา Server ต้องไปเปิดหาใน DB ทุกครั้ง ว่านี่คือใคร!

💡 ระบบนี้เรียกว่า "Stateful" เพราะ Server ต้องจดจำสถานะ (State) ของผู้ใช้ทุกคนไว้ในหน่วยความจำหรือฐานข้อมูลของตัวเอง

💥 ปัญหาของระบบ Session-Based

  • Scalability ยาก: ถ้าเรามี Server 5 ตัว (Load Balancer) User ล็อกอินที่ Server A แต่ครั้งต่อไป Request วิ่งไป Server B... Server B จะไม่รู้จัก User คนนี้ (เพราะ Session อยู่ที่ A)
  • ภาระฐานข้อมูล (Overhead): ต้องอ่าน Database/Redis ทุกๆ Request ยิ่งคนใช้เยอะ DB ยิ่งทำงานหนัก
  • CORS & Mobile Apps: การจัดการ Cookie ข้ามโดเมน หรือใน Mobile App มีความซับซ้อน
❓ คำถามชวนคิด

ถ้าเราต้องการรองรับผู้ใช้ 1 ล้านคนพร้อมกัน การเก็บ Session 1 ล้าน Record และต้อง Query มันในทุกๆ วินาที จะเกิดอะไรขึ้นกับระบบ?

🚀 ทางออก: JWT และระบบแบบ Stateless

"ถ้างั้น ก็ไม่ต้องให้ Server จำอะไรเลยสิ!" นี่คือคอนเซปต์ของ Stateless

✅ JWT (JSON Web Token) ทำหน้าที่เสมือน "บัตรประชาชน" หรือ "พาสปอร์ต"

1. User ล็อกอินสำเร็จ Server ออกบัตร (JWT) ให้ User ไปถือไว้

2. ข้อมูลว่า User คือใคร สิทธิ์อะไร ถูกเขียนติดไว้บนบัตรเลย

3. พอ User จะใช้งาน ก็แค่ยื่นบัตร (Token) ให้ Server

4. Server แค่ตรวจสอบ "ลายเซ็นบนบัตร" ว่าเป็นของจริงและไม่หมดอายุ ก็ให้ผ่านได้เลย โดยไม่ต้องไปค้น Database!

ผ่าโครงสร้าง JWT (Anatomy of JWT) 🧬

เบื้องหลังสตริงยาวๆ มันซ่อนอะไรไว้ข้างในบ้าง?

ชำแหละโครงสร้าง JWT

JWT คือ String ยาวๆ ที่ถูกคั่นด้วยเครื่องหมายจุด (.) แบ่งออกเป็น 3 ส่วน:

1. Header (สีแดง)

บอกประเภท Token (JWT) และ Algorithm ที่ใช้เข้ารหัสลายเซ็น (เช่น HS256)

2. Payload (สีม่วง)

เก็บข้อมูลผู้ใช้ (Claims) เช่น User ID, Role, วันหมดอายุ

3. Signature (สีฟ้า)

ลายเซ็นดิจิทัล ใช้ยืนยันว่าข้อมูลไม่ได้ถูกดัดแปลง

🚨 ระวัง! ถอดรหัส (Decode) ไม่ใช่ เข้ารหัสลับ (Decrypt)

ข้อมูลใน Header และ Payload ไม่ได้ถูกเข้ารหัสลับ (Not Encrypted)

มันถูกแค่แปลงฟอร์แมตให้อยู่ในรูป Base64Url เท่านั้น!

แปลว่า "ใครก็ตาม" ที่ได้ Token นี้ไป สามารถเอาไป Decode อ่านข้อมูลข้างในได้อย่างง่ายดายผ่านเว็บเช่น jwt.io

⚠️ กฎเหล็กของ JWT

ห้าม!!! ใส่ข้อมูลที่เป็นความลับ เช่น Password, บัตรเครดิต, หรือข้อมูลส่วนบุคคลที่อ่อนไหว ลงใน Payload เด็ดขาด ให้ใส่แค่ ID, Role หรือข้อมูลที่เปิดเผยได้เท่านั้น

❓ คำถาม: ถ้าใครก็อ่านได้ แล้วมันปลอดภัยยังไง?

ความปลอดภัยของ JWT ไม่ได้อยู่ที่ "การซ่อนข้อมูล" แต่อยู่ที่ "ความสมบูรณ์ของข้อมูล" (Data Integrity)

เวทมนตร์อยู่ที่ส่วนที่ 3 คือ Signature

สูตรการสร้าง Signature:
HMACSHA256( base64UrlEncode(header) + "." + base64UrlEncode(payload), SECRET_KEY )

  • การจะสร้าง Signature ที่ถูกต้องได้ ต้องรู้ SECRET_KEY (ซึ่งเก็บไว้ที่ Server เท่านั้น)
  • ถ้า Hacker เอา Token ไปแอบแก้ Payload (เช่น เปลี่ยน Role=User เป็น Role=Admin)
  • เมื่อส่งกลับมา Server จะเอา Header กับ Payload ใหม่ มาคำนวณ Signature อีกครั้ง
  • ผลลัพธ์: Signature ใหม่จะไม่ตรงกับของเดิม! (เพราะ Hacker ไม่มี Secret) Server จะเตะ Request นั้นทิ้งทันที! (401 Unauthorized)

Part 3

Algorithm ที่ใช้ Sign Token 🔐

การเลือกกุญแจล็อคความปลอดภัย: Symmetric vs Asymmetric

🔑 HS256 vs RS256 ต่างกันอย่างไร?

ใน Header เราจะเห็นค่า "alg": "HS256" นี่คือ Algorithm หลักๆ มี 2 ประเภทที่นิยม

Symmetric (เช่น HS256)

ใช้ Secret Key 1 ดอก ถือกุญแจดอกเดียวกันทั้งตอน "สร้าง(Sign)" และตอน "ตรวจ(Verify)"

✅ เร็ว ง่าย เหมาะกับระบบเล็ก (Monolith)

❌ ถ้า Service อื่นจะตรวจ Token ต้องแจกกุญแจดอกนี้ไปให้ด้วย (เสี่ยงกุญแจหลุด)

Asymmetric (เช่น RS256)

ใช้ Key Pair 2 ดอก
- Private Key ใช้ "สร้าง(Sign)"
- Public Key ใช้ "ตรวจ(Verify)"

✅ ปลอดภัยมาก Service อื่นเอา Public Key ไปตรวจได้โดยไม่ต้องห่วงความลับหลุด

Part 4

การทำงานและ Flow ของ JWT 🔄

วงจรชีวิตตั้งแต่ Login ยัน Logout

ขั้นตอนที่ 1: การขอ Token (Authentication)

1. Client
(POST /login + User/Pass)
2. Auth Service
(Verify DB -> Sign JWT)
3. Client
(รับ JWT ไปเก็บไว้)

1. ผู้ใช้กรอก Username / Password

2. Server ตรวจสอบกับ Database ถ้ารหัสถูก จะทำการ Sign JWT (ใช้ Secret Key)

3. Server ส่ง String JWT กลับไปให้ Client (Browser/Mobile) เก็บไว้ (เช่น ใน LocalStorage หรือ Cookie)

ขั้นตอนที่ 2: การใช้งาน Token (Authorization)

1. Client
(GET /api/data + Header)
2. API Gateway/Service
(Verify Signature)
3. Response
(Return Data / 401)

1. ทุกครั้งที่ Client จะดึงข้อมูล ต้องแนบ JWT ไปด้วยใน HTTP Header:
Authorization: Bearer

2. Server รับ Request หยิบ Token มาตรวจสอบ (Verify Signature & Expiration Date)

3. ถ้าตรวจสอบผ่าน (Valid) Server จะให้บริการดึงข้อมูลส่งกลับไป

Part 5

JWT ในโลก Microservices 🏗️

ทำไมมันถึงกลายเป็นมาตรฐาน (De facto standard)?

ทำไม Microservices ถึงรัก JWT? 💖

ในสถาปัตยกรรม Microservices ระบบถูกซอยเป็นหลายๆ Service (Order, Product, Payment)

  • ถ้าใช้ Session: ทุก Service ต้องต่อท่อไปหา Redis/Database ตัวเดียวกันเพื่อเช็คว่า User ล็อกอินหรือยัง (เกิด Bottleneck & Coupling)
  • เมื่อใช้ JWT:
    • Auth Service เป็นคนออกบัตร (Sign)
    • Service อื่นๆ แค่ "ตรวจบัตร" (Verify) ด้วย Library ในตัวมันเอง หรือใช้ API Gateway ตรวจด่านหน้า
    • ระบบทำงานอิสระต่อกัน (Decoupled) สเกลได้ไม่จำกัด!

[Auth Service] ➔ (Issues JWT)

↓

[Client] ➔ [API Gateway] (Verifies JWT)

↙     ↓     ↘

[Svc A]   [Svc B]   [Svc C]

(Services ได้รับ User ID ทันที ไม่ต้องถาม DB!)

Part 6

ข้อพึงระวัง & Security Best Practices 🛡️

ไม่มีอะไรสมบูรณ์แบบ JWT ก็มีจุดอ่อนที่ต้องระวัง

ชั่งน้ำหนัก: ข้อดี vs ข้อเสีย ของ JWT ⚖️

✅ ข้อดี (Pros)
  • Stateless & Scalable: Server ไม่ต้องจำ State
  • Cross-Domain: ทำงานข้ามโดเมนได้สบาย (CORS friendly กว่า Session Cookie)
  • Performance: ลด Load ของ Database ได้มหาศาล
  • Mobile Ready: ใช้งานกับ Mobile App ได้ง่าย
❌ ข้อเสีย (Cons)
  • ขนาดใหญ่ (Payload Size): ยิ่งใส่ข้อมูลเยอะ Token ยิ่งยาว เปลือง Bandwidth (เพราะต้องส่งทุก Request)
  • Invalidation ทำได้ยาก: "ถ้าออก Token ไปแล้ว จะยกเลิกกลางคันยังไง?" (นี่คือปัญหาใหญ่สุดของ JWT)
  • ข้อมูลอ่านได้ (Not Encrypted): เสี่ยงถ้าใส่ข้อมูลผิดประเภท

🚨 ปัญหาโลกแตก: การยกเลิก Token (Invalidation)

เนื่องจาก JWT ตรวจสอบตัวเองได้ (Stateless) ถ้า User กด Logout หรือ โดนแบน...

ปัญหา: Token เก่าที่ออกไปแล้ว ยังคง Valid อยู่จนกว่าจะถึงเวลาหมดอายุ (Expiration) ที่ตั้งไว้!

💡 วิธีแก้ปัญหา (Mitigation Strategies)
  1. ตั้งอายุสั้นๆ (Short Lived Token): ให้ Access Token หมดอายุไวๆ (เช่น 15 นาที) เพื่อลดหน้าต่างความเสี่ยง
  2. ใช้ Refresh Token: เมื่อ Access Token หมดอายุ ค่อยใช้ Refresh Token (ซึ่งเก็บใน DB) ไปขอใหม่ ถ้า User โดนแบน ก็ลบ Refresh Token ใน DB ทิ้ง!
  3. Token Blacklist: ยอมทำ Stateful นิดๆ โดยเก็บ Token ที่ถูกสั่ง Logout ไว้ใน Redis ชั่วคราวจนกว่าจะหมดอายุ (แลกมากับ Performance ที่ลดลง)

เก็บ Token ไว้ที่ไหนดี? (Front-end Storage) 🗄️

ความเสี่ยงหลัก: Hacker ขโมย Token ของผู้ใช้ไปสวมรอย

❌ Local Storage / Session Storage

เข้าถึงได้ด้วย JavaScript (localStorage.getItem('token'))

เสี่ยงต่อ XSS (Cross-Site Scripting): ถ้าเว็บเรามีช่องโหว่ให้รัน JS แปลกปลอมได้ Hacker จะดูด Token ขโมยออกไปได้ทันที!

✅ HttpOnly Cookie

ตั้งค่าคุกกี้เป็น HttpOnly และ Secure

JavaScript ของ Browser จะมองไม่เห็นและอ่านไม่ได้ ป้องกัน XSS ได้ชะงัดนัก!

(แต่ต้องไประวังเรื่อง CSRF แทน ซึ่งแก้ได้ด้วย SameSite cookie)

💻 ตัวอย่างโค้ด (Node.js) - การสร้าง Token

เราใช้ Library อย่าง jsonwebtoken ในการสร้าง (Sign)

const jwt = require('jsonwebtoken');
const SECRET_KEY = process.env.JWT_SECRET; // ห้ามฮาร์ดโค้ด!

function login(req, res) {
  // ... ตรวจสอบ User/Pass สำเร็จ ...
  const payload = {
    userId: 123,
    role: 'admin'
  };

  // สร้าง Token (Sign)
  const token = jwt.sign(payload, SECRET_KEY, { expiresIn: '1h' });
  
  res.json({ token: token });
}

💻 ตัวอย่างโค้ด (Node.js) - การตรวจ Token (Middleware)

เราเขียน Middleware ดัก Request ก่อนเข้าถึงข้อมูลจริง

function authenticateToken(req, res, next) {
  const authHeader = req.headers['authorization'];
  const token = authHeader && authHeader.split(' ')[1]; // แยกว่า Bearer 

  if (!token) return res.sendStatus(401); // ไม่มี Token เตะออก

  // ตรวจสอบ (Verify)
  jwt.verify(token, SECRET_KEY, (err, user) => {
    if (err) return res.sendStatus(403); // Token ปลอมหรือหมดอายุ
    req.user = user; // เอา payload แปะไว้ให้ Service ถัดไปใช้
    next(); // ให้ผ่าน!
  });
}

⚠️ จุดผิดพลาดบ่อย: บางคนเผลอใช้ jwt.decode() แทน jwt.verify() ซึ่งมันจะแค่อ่านข้อมูล ไม่ออกแรงตรวจสอบ Signature (โดนแฮกแน่นอน!)

🔧 สรุปการใช้งานในชีวิตจริงของสถาปนิกซอฟต์แวร์

  • อย่าสร้างระบบ Auth เองถ้าไม่จำเป็น: ให้ใช้บริการภายนอก เช่น Keycloak, Auth0, Firebase Auth หรือ AWS Cognito ซึ่งเขาจัดการเรื่อง JWT, Refresh Token และ Security ให้หมดแล้ว (ตามมาตรฐาน OAuth 2.0 / OpenID Connect)
  • API Gateway: ให้ API Gateway (เช่น Kong, AWS API Gateway) เป็นคนทำหน้าที่ Verify JWT เพื่อลดภาระของ Microservices ด้านใน

โจทย์: JWT Debugger

  1. ให้นักศึกษาเข้าไปที่เว็บไซต์ https://jwt.io
  2. ลองเปลี่ยนข้อมูลในกล่องสีม่วง (Payload) สังเกตว่าเกิดอะไรขึ้นกับ Token ทางซ้าย?
  3. ลองเปลี่ยน Secret Key สีฟ้าด้านล่างขวา สังเกตการณ์เปลี่ยนแปลงของ Signature

📌 สรุป (Key Takeaways)

1. JWT (JSON Web Token) เป็นเครื่องมือที่ทำให้ระบบ Authentication เป็นแบบ Stateless (Server ไม่ต้องจำ State)

2. มี 3 ส่วน: Header (ประเภท+อัลกอริทึม), Payload (ข้อมูล-ห้ามใส่ความลับ!), Signature (ลายเซ็นกันปลอมแปลง)

3. Signature ทำงานได้เพราะ Hacker ไม่มี Secret Key / Private Key ของเรา

4. Microservices นิยมใช้มากเพราะลดปัญหาคอขวดที่ฐานข้อมูล

5. ต้องระวังเรื่อง Storage (XSS) และการทำ Token Invalidation (Refresh Token)
