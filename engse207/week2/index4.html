คำตอบอยู่ที่แนวคิด Quality Attributes & Architectural Drivers

Quality Attributes คืออะไร?

Quality Attributes = คุณสมบัติที่ใช้วัด “คุณภาพ” ของระบบซอฟต์แวร์ นอกเหนือจากฟังก์ชัน

  • มักถูกเรียกว่า Non-Functional Requirements หรือ “-ilities”
  • เช่น ความเร็ว, ความปลอดภัย, ความยืดหยุ่น, ความพร้อมใช้งาน
  • เป็นตัวกำหนดว่า “ผู้ใช้รู้สึกว่าระบบดีไหม” ไม่ใช่แค่ “ทำงานได้หรือเปล่า”
คิดง่ายๆ: Functional = ระบบทำอะไรได้บ้าง
Quality Attributes = ระบบทำ “ได้แค่ไหน” และ “ดีแค่ไหน”

ทำไมต้องสนใจ Quality Attributes?

  • เป็นตัว กำหนดสถาปัตยกรรม ที่เหมาะสม
  • ใช้เป็น เกณฑ์ตัดสินใจ ระหว่างทางเลือกต่างๆ
  • ช่วย สื่อสารกับ Stakeholders ให้เห็นภาพเดียวกัน
  • ใช้เป็น เกณฑ์วัดความสำเร็จของระบบ

6 Quality Attributes หลัก

  • Performance – เร็ว / ตอบสนองทันเวลา
  • Scalability – ขยายรองรับผู้ใช้/โหลดที่เพิ่มขึ้นได้
  • Availability – พร้อมใช้งาน ไม่ล่มง่าย
  • Security – ปลอดภัย ปกป้องข้อมูล
  • Modifiability – แก้ไข/เพิ่มฟีเจอร์ง่าย
  • Usability – ใช้งานง่าย ผู้ใช้เข้าใจ

แต่ละตัวมีผลโดยตรงต่อการเลือกโครงสร้างระบบ เช่น Monolith, Microservices, Caching, Replication ฯลฯ

Performance (ประสิทธิภาพ)

วัดความสามารถของระบบในการตอบสนองต่อคำขอภายในเวลาที่กำหนด

  • Response Time – เวลาตอบสนองต่อ 1 request
  • Throughput – จำนวน requests/second ที่รองรับได้
  • Latency – ความหน่วงของเครือข่าย
  • Resource Utilization – การใช้ CPU, RAM, Disk, Network

ตัวอย่างจากระบบจริง

  • เกมออนไลน์ Battle Royale – ต้องการ latency ต่ำ < 50ms
  • Flash Sale E-Commerce – รองรับ traffic หลักแสน request/วินาที
  • Video Streaming – โหลดไว ไม่กระตุก Buffer time น้อย
สถาปัตยกรรมที่ช่วย Performance:
Caching, Load Balancer, Database Indexing, Asynchronous Processing ฯลฯ

Scalability (ขยายได้)

ความสามารถของระบบในการรองรับโหลดที่เพิ่มขึ้นโดยประสิทธิภาพไม่ตกมาก

  • Vertical Scaling – อัปเกรดเครื่องเดิม (เพิ่ม CPU / RAM)
  • Horizontal Scaling – เพิ่มจำนวนเครื่อง/instances
  • Elasticity – ขยาย/หดตามโหลดอัตโนมัติ (Auto-scaling)

ตัวอย่าง

  • Messaging app ที่รองรับผู้ใช้งานหลายสิบล้านคน
  • Music streaming ที่จำนวนผู้ใช้ active เพิ่มขึ้นทุกปี
  • ระบบสั่งอาหาร/เดลิเวอรีช่วงฝนตก – โหลดพุ่งกระทันหัน
สถาปัตยกรรมที่ช่วย Scalability:
Stateless services, Microservices, Sharding, Kubernetes, Message Queue

Availability (พร้อมใช้งาน)

สัดส่วนเวลาที่ระบบให้บริการได้ตามปกติ ไม่ล่ม

ระดับ Uptime Downtime/ปี
Two Nines99%~3.65 วัน
Three Nines99.9%~8.76 ชม.
Four Nines99.99%~52 นาที
Five Nines99.999%~5 นาที

ตัวอย่างระบบที่ต้องการ High Availability

  • Mobile Banking / Payment
  • Hospital Information System (HIS)
  • Email / Communication Platform
เทคนิคด้านสถาปัตยกรรม:
Redundancy, Failover, Health checks, Multi-region deployment, Circuit Breaker

Security (ความปลอดภัย)

ปกป้องระบบและข้อมูลจากการเข้าถึง/ใช้งานที่ไม่ได้รับอนุญาต

  • Confidentiality – ข้อมูลลับ ไม่รั่วไหล
  • Integrity – ข้อมูลถูกต้อง ไม่ถูกแก้ไข
  • Availability – ระบบพร้อมให้บริการแม้มีการโจมตี

เกี่ยวข้องกับ Authentication, Authorization, Encryption, Auditing เป็นต้น

ตัวอย่าง

  • Internet Banking – ต้องมี 2FA, logging ทุก transaction
  • ระบบภาครัฐ – ต้อง comply PDPA, เก็บ log ตามกฎหมาย
  • HR/Payroll – ต้องควบคุมการเข้าถึงด้วย Role-based access
สถาปัตยกรรมที่ช่วย Security:
API Gateway, OAuth2/JWT, TLS, WAF, Zero-trust, Centralized logging & alerting

Modifiability (แก้ไขง่าย)

ระบบสามารถเปลี่ยนแปลง/เพิ่มฟีเจอร์ได้เร็ว โดยไม่พังส่วนอื่น

  • ต้องแก้โค้ดน้อยส่วนเมื่อมี requirement ใหม่
  • เวลาพัฒนา feature ใหม่สั้นลงเรื่อยๆ
  • แยกส่วนชัดเจน (Separation of Concerns, Loose Coupling)

เชื่อมโยงกับแนวคิด Clean Architecture, Modular, Plugin-based

Usability (ใช้งานง่าย)

ทั้งฝั่งผู้ใช้, นักพัฒนา และ Ops ต้องใช้งานระบบได้อย่างไม่ปวดหัว

  • UI/UX เข้าใจง่าย · มี feedback ชัดเจน
  • Developer มี API/Docs ที่ดี
  • Ops มี dashboard, monitoring, error message ที่เข้าใจได้

Trade-offs: ไม่ได้ทุกอย่างพร้อมกัน

การออกแบบสถาปัตยกรรม = ศิลปะการ “เลือก” ว่าจะเน้น Quality Attribute ตัวไหน

  • เพิ่ม Security → Performance มักลดลง
  • เพิ่ม Availability → อาจต้องยอมรับ Eventual Consistency
  • เพิ่ม Flexibility/Modifiability → มี Overhead เพิ่ม, Performance ลด
  • Scale ใหญ่ขึ้น → ค่าใช้จ่ายสูงขึ้น
Key Insight: เราต้องรู้ Priority ก่อนออกแบบ ไม่อย่างนั้นจะหลงทาง

ตัวอย่าง Trade-offs ที่พบบ่อย

Trade-off #1
Performance ↔ Security
  • Validate token ทุก request → ปลอดภัยแต่ช้ากว่า
  • ใช้ HTTPS แทน HTTP → มี TLS overhead แต่ปกป้องข้อมูล
  • ระบบธนาคาร: เลือก Security ก่อน
Trade-off #2
Consistency ↔ Availability
  • CAP theorem ในระบบ distributed
  • Banking: เลือก strong consistency
  • Feed social: เลือก availability + eventual consistency
Trade-off #3
Scalability ↔ Cost
  • Auto-scaling, multi-region = ต้นทุน infra สูง
  • Startup มักเริ่ม simple แล้วค่อย scale
Trade-off #4
Modifiability ↔ Performance
  • Microservices: ยืดหยุ่น แต่มี network overhead
  • Monolith: เร็วกว่า แต่เปลี่ยนยาก

Quality Attribute Scenario คืออะไร?

แทนที่จะบอกแค่ว่า “ระบบต้องเร็ว / ปลอดภัย” เราเขียนเป็น สถานการณ์ที่วัดผลได้จริง

  • ระบุว่า “ใครทำอะไร กับส่วนไหน ของระบบ ในสถานการณ์ใด”
  • ระบุการตอบสนองของระบบ
  • ระบุเกณฑ์วัดผล (ต้องมีตัวเลข!)
Use case vs QA Scenario
Use case = ระบบทำ “อะไร”
QA Scenario = ระบบมีคุณภาพ “แค่ไหน”

ประโยชน์

  • ทำให้ requirement ชัดเจนและทดสอบได้
  • ใช้เปรียบเทียบสถาปัตยกรรมหลายแบบ
  • ใช้เป็น checklist ตอนทำ Performance / Security test

โครงสร้าง Scenario 6 ส่วน

  • 1. Source – ใคร/อะไรเป็นตัวกระตุ้น
  • 2. Stimulus – เหตุการณ์อะไรเกิดขึ้น
  • 3. Artifact – ส่วนไหนของระบบได้รับผลกระทบ
  • 4. Environment – สภาพแวดล้อม ณ ตอนนั้น
  • 5. Response – ระบบตอบสนองอย่างไร
  • 6. Response Measure – วัดผลความสำเร็จอย่างไร

Architectural Drivers คืออะไร?

คือ ชุด requirement สำคัญ ที่ “ขับเคลื่อน” การออกแบบสถาปัตยกรรม

  • Functional Requirements – ระบบต้องทำอะไร
  • Quality Attributes – ต้องดีแค่ไหน ในด้านต่างๆ
  • Constraints – ข้อจำกัดที่เปลี่ยนไม่ได้ง่ายๆ
  • Assumptions – สมมติฐานที่เราใช้วางแผน

1. Functional Requirements

สิ่งที่ระบบต้องทำได้ · Features · Use cases

  • สร้าง/แก้ไข/ลบ ข้อมูลสำคัญ
  • Workflow หลักของผู้ใช้แต่ละกลุ่ม
  • Integration กับระบบภายนอก

ตัวอย่าง: Task Board System

  • สร้าง/จัดการ Board & Column (To-Do, In-Progress, Done)
  • สร้าง/ลากวาง Task card + Assign ให้สมาชิกทีม
  • Real-time notification เมื่อมีการอัปเดต
  • Attach files, comment ในแต่ละ task
  • Export รายงานเป็น PDF หรือ Excel

2. Quality Attributes (Drivers หลัก)

กำหนด “คุณภาพ” ที่สถาปัตยกรรมต้องตอบให้ได้

  • Performance – ต้องเร็วแค่ไหน?
  • Scalability – รองรับผู้ใช้/ข้อมูลเท่าไหร่?
  • Availability – ล่มได้กี่นาที/ปี?
  • Security – ต้องผ่านมาตรฐานอะไร?
  • Modifiability & Usability ฯลฯ

ตัวอย่าง Scenario สั้นๆ

  • โหลด board ที่มี 200 tasks ใน < 1.5 วินาที (95th percentile)
  • รองรับทีมขนาด 1,000 users พร้อมกันโดยระบบไม่ล่ม
  • Audit log เก็บย้อนหลังอย่างน้อย 1 ปี

นี่คือ input สำคัญของ Workshop & Lab วันนี้

3. Constraints (ข้อจำกัด)

  • Technical – ต้องใช้ tech stack / cloud / DB ที่กำหนด
  • Time – เส้นตาย Project, ช่วงเวลา release
  • Budget – ต้นทุน infra / license
  • Legal/Policy – PDPA, data residency, audit log

4. Assumptions (สมมติฐาน)

  • จำนวน peak users ไม่เกิน X
  • อินเทอร์เน็ตผู้ใช้ส่วนใหญ่เป็น 4G/Wi-Fi
  • Third-party API ที่เชื่อมมี uptime ~99%
  • ขนาดข้อมูลปีแรกไม่เกิน 10GB เป็นต้น

ถ้า Assumption ผิด → สถาปัตยกรรมอาจไม่พอ ต้องปรับใหม่

ตัวอย่าง Architectural Drivers · Task Board System

Functional
  • Kanban board, drag-and-drop tasks
  • Real-time notifications
  • File attachment & comments
  • Search / filter / report
Quality Attributes
  • Response time board view < 1.5s
  • Support 1,000 team users
  • Uptime ≥ 99% ในเวลางาน
  • Role-based access control
Constraints
  • ใช้ Node.js + React
  • Deploy บน Cloud (เช่น AWS/GCP)
  • Budget infra ไม่เกิน xx บาท/เดือน

  • Team size 10–100 คน
  • Peak concurrent users ≤ 200 ใน Phase แรก
  • Board ละ ≤ 1,000 tasks

Artifact[ส่วนใดของระบบ]
Environment[สภาวะแบบใด]
Response[ระบบควรทำอะไร]
Response Measure[ตัวเลขวัดผล]

เน้น Response Measure ให้วัดได้ – เช่น < 2 วินาที, 95% ของ requests ฯลฯ

Activity 3 · วิเคราะห์ Trade-offs

เวลา: ~30 นาที

  • ดูว่า Quality Attributes ที่เลือกมามีคู่ไหนขัดแย้งกันบ้าง
  • เขียนอธิบาย Trade-off อย่างน้อย 1 คู่
  • ตัดสินใจว่าจะเน้นตัวไหนมากกว่าใน context ของ Task Board

ตัวอย่าง Pair ให้คิด

  • Performance vs Security
  • Scalability vs Cost
  • Modifiability vs Performance

คำถาม: ถ้า requirement บอกว่า “ต้อง real-time มากๆ” แต่ budget น้อยมาก คุณจะออกแบบสถาปัตยกรรมอย่างไร?

Activity 4 · Presentation & Peer Review

เวลา: ~20 นาที

  • แต่ละกลุ่มนำเสนอ Top 3 Scenarios + Trade-offs ที่สำคัญ
  • เพื่อนกลุ่มอื่นให้ feedback ตาม checklist
เป้าหมาย: ฝึกอธิบายเหตุผลเชิงสถาปัตยกรรมให้ทีมอื่นเข้าใจ

Peer Review Checklist (ย่อ)

  • Scenario ครบ 6 ส่วนหรือไม่
  • Response Measure วัดได้จริง?
  • Trade-offs analysis มีเหตุผล?
  • Priority ของ Attributes สอดคล้องกับระบบไหม?

ใช้คะแนน Peer Review เป็น input ปรับปรุงเอกสารก่อนทำ Lab

Lab · Architectural Drivers Document

งานใหญ่ของสัปดาห์นี้ – สร้างเอกสาร Architectural Drivers สำหรับโปรเจกต์ของทีม

  • เลือก Domain ระบบที่สนใจ (E-Learning, Food delivery, IoT, Social, Healthcare, Hotel booking ฯลฯ)
  • เขียน System overview + Functional requirements
  • ระบุ Quality Attributes + Scenarios, Constraints, Assumptions
  • จัดลำดับความสำคัญและ Trade-offs

กำหนดส่ง & รูปแบบ

  • ส่งภายใน 1 สัปดาห์ (รูปแบบเอกสารตาม template)
  • ทำเป็นกลุ่มเดียวกับ Project หลักของวิชา (ถ้ามี)
  • ใช้ภาษาไทยหรืออังกฤษก็ได้ แต่ต้องชัดเจนและวัดผลได้

เอกสารนี้จะถูกใช้ต่อในสัปดาห์ถัดไปตอนออกแบบสถาปัตยกรรม

โครงสร้างเอกสาร Lab (ส่วนที่ 1–3)

  1. System Overview (1 หน้า)
    อธิบายระบบ, ปัญหาที่แก้, Target users, Top 10 features
  2. Functional Requirements (1–2 หน้า)
    ระบุ FR อย่างน้อย 15 ข้อ แบ่งตามหมวด (User Management, Core, Notification, Reporting ฯลฯ)
  3. Quality Attributes & Scenarios (2–3 หน้า)
    เลือก 6 Attributes หลัก เขียน Scenario ละ 1 อัน พร้อมตาราง 6 ส่วน + คำอธิบาย

Tip: เริ่มจาก Use cases สำคัญ → สร้าง FR → ค่อยหา Quality Attributes ที่สำคัญที่สุด

โครงสร้างเอกสาร Lab (ส่วนที่ 4–6)

  1. Constraints (1 หน้า)
    แยกเป็น Technical, Time, Budget, Legal/Policy
  2. Assumptions (0.5 หน้า)
    ระบุสมมติฐานสำคัญอย่างน้อย 5 ข้อ
  3. Priority & Trade-offs (1 หน้า)
    • ตารางจัดอันดับ Quality Attributes 1–6 พร้อมเหตุผล
    • วิเคราะห์ Trade-off อย่างน้อย 1 คู่ พร้อมการตัดสินใจ

เอกสารควรอ่านง่าย มีหัวข้อชัดเจน ใช้ bullet / ตาราง ช่วยให้ Architecture team ทำงานต่อได้สะดวก

เกณฑ์การให้คะแนน (100 คะแนน)

System Overview10
Functional Requirements20
Quality Attribute Scenarios (6 ชุด)30
Constraints10
Assumptions5
Priority & Trade-offs15
รูปแบบเอกสาร/ความอ่านง่าย10

คำแนะนำ

  • Response measure ต้องมีตัวเลข (ไม่ใช้คำว่า “เร็ว” เฉยๆ)
  • Trade-offs ต้องอธิบายเหตุผล ไม่ใช่แค่ list
  • พยายามอ้างอิงตัวอย่างจากระบบจริง/คู่แข่ง
  • เขียนให้เพื่อนทีมอื่นอ่านแล้วเข้าใจระบบของเราได้

เชื่อมกับ CLO ของรายวิชา

CLO3 (K)
อธิบาย Quality Attributes ได้
  • รู้จัก Performance, Scalability, Availability, Security, Modifiability, Usability
  • เข้าใจ Scenario 6 ส่วน
  • มองเห็น Trade-offs ที่สำคัญ
CLO5 & CLO7 (C)
วิเคราะห์ & ประเมินสถาปัตยกรรม
  • ระบุ Architectural Drivers จาก requirements
  • จัดลำดับความสำคัญของคุณภาพ
  • เปรียบเทียบทางเลือก โดยมีเหตุผลรองรับ

Lab ของสัปดาห์นี้จะเป็นหลักฐานสำคัญสำหรับ CLO เหล่านี้ใน มคอ.3

สรุปวันนี้

  • รู้จัก 6 Quality Attributes หลัก + ตัวอย่างจากระบบจริง
  • เรียนรู้การเขียน Quality Attribute Scenarios แบบวัดผลได้
  • เข้าใจว่าทำไมต้องมี Trade-offs และวิธีคิดเวลาต้องเลือก
  • เห็นภาพของ Architectural Drivers ว่าเป็น “เข็มทิศ” ของการออกแบบ

Key Takeaway: ก่อนออกแบบสถาปัตยกรรมทุกครั้ง ต้องตอบได้ ว่า “Quality Attributes อะไรสำคัญที่สุด & เราพร้อมแลกอะไรบ้าง”

สัปดาห์หน้า: Architectural Styles & Patterns (Part 1)

  • Monolithic, Layered, Client-Server, N-Tier, Pipe-and-Filter
  • ดูว่าคุณภาพแต่ละแบบเหมาะกับ Style ไหน
  • เริ่ม map Architectural Drivers ของทีม → เลือก Style เบื้องต้น

เตรียมตัว: เลือก Domain สำหรับ Lab ให้เรียบร้อย และลองคิดคร่าวๆ ว่าระบบของคุณควรเน้น Quality Attribute ใดเป็นอันดับ 1–3

Q & A · คุยกันก่อนจบคาบ

  • มี Quality Attribute ไหนที่ยังงงอยู่หรืออยากได้ตัวอย่างเพิ่ม?
  • ระบบที่ทุกคนใช้อยู่ประจำ (เกม/แอป/เว็บ) เน้นคุณภาพด้านไหน?
  • คิดว่า Project จบของตัวเองควรเน้น Attribute อะไร 3 อันดับแรก?

Discussion Prompt: ให้แต่ละคนลองแชร์ 1 ระบบที่ชอบมาก และระบุ 1 Quality Attribute ที่คิดว่าทำได้ดีสุด (เช่น “Netflix – Usability/UX ดีมาก”)
