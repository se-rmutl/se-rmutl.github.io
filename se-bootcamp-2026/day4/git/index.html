ทำไม Software Engineer ต้องใช้ GitHub?

ในฐานะวิศวกรซอฟต์แวร์ GitHub คือ "พาสปอร์ต" และ "หน้าตา" ของเราในโลกการทำงานยุคปัจจุบัน

  • Evidence of Learning: ใช้เป็นหลักฐานแสดงการเรียนรู้จริงที่โปร่งใสและตรวจสอบย้อนหลังได้
  • Industry Standard: บริษัทเทคโนโลยีชั้นนำทั่วโลกใช้ Git/GitHub ในการทำงานร่วมกันเป็นทีม
  • Continuous Tracking: บันทึกและสะสมประวัติการพัฒนา (Commit History) ของตัวเองตั้งแต่วันแรก

Concept 1: Version Control System (VCS)

ลองจินตนาการถึงตอนที่คุณทำรายงานกลุ่มในอดีต...

วิธีแบบเก่า (ยุคอนาล็อก)

มีไฟล์ชื่อ รายงาน_final.docx, รายงาน_final_แก้ครั้งที่2.docx, รายงาน_final_จริงจังอันนี้สุดท้าย.docx เต็มหน้าจอ เกิดความสับสนอย่างรุนแรง

วิธีของวิศวกรซอฟต์แวร์

ใช้ไฟล์เดียวชื่อเดิมตลอดเวลา แต่ใช้ระบบ VCS คอยเก็บ "Timeline ประวัติ" ไว้เบื้องหลังอย่างละเอียด สามารถกดย้อนเวลากลับไปดูโค้ดเก่าได้ทุกเมื่อ

Concept 2: ความต่างระหว่าง Git และ GitHub

คุณลักษณะ Git (Local Engine) GitHub (Cloud Platform)
มันคืออะไร? ซอฟต์แวร์/เครื่องมือระบบ Version Control แพลตฟอร์มเว็บไซต์บน Cloud สำหรับฝากไฟล์
ทำงานอยู่ที่ไหน? ทำงานในเครื่องคอมพิวเตอร์ของเรา ทำงานบนอินเทอร์เน็ต
สถานะเน็ตเวิร์ก? ทำงานแบบ Offline ได้ ไม่ต้องใช้เน็ต ต้องต่อ Online เสมอเพื่อเข้าถึงข้อมูล
อุปมาอุปไมย สมุดบันทึกส่วนตัวในกระเป๋าของคุณ หอสมุดกลางของเมืองที่แชร์ให้คนอื่นมาอ่านได้

ถามก่อนเฉลย: ตรรกะการ Sync ข้อมูล

"ถ้าเราบันทึกประวัติ (Commit) งานในเครื่องคอมพิวเตอร์ของเราสำเร็จแล้ว แต่ไฟฟ้าที่บ้านดับทันที..."

อาจารย์และเพื่อนๆ จะสามารถมองเห็นความเปลี่ยนแปลงนั้นบนหน้าเว็บ GitHub ของเราได้เลยหรือไม่?

คำตอบ: มองไม่เห็น! (No)

เนื่องจากคำสั่ง git commit บันทึกข้อมูลลงสู่ฐานข้อมูล Git ภายในเครื่อง (Local) ของเราเท่านั้น ข้อมูลจะยังไม่เดินทางไปหา GitHub จนกว่าเราจะใช้คำสั่ง git push ร่วมกับการต่ออินเทอร์เน็ตครับ

Concept 3: Repository คืออะไร?

คำว่า Repository (หรือเรียกสั้นๆ ว่า Repo) ในบริบทของวิศวกรรมซอฟต์แวร์ หมายถึง โฟลเดอร์เก็บโปรเจกต์โปรแกรมที่มีระบบ Git คอยจับตาดูอยู่

โฟลเดอร์ปกติ + สั่งเริ่มใช้ Git = Repository

ภายใน Repo จะมีโฟลเดอร์ลับชื่อ .git ถูกสร้างขึ้นมาเพื่อเก็บ Metadata ประวัติทั้งหมด ย้อนหลังตั้งแต่จุดเริ่มต้นโปรเจกต์

Developer Tools: เครื่องมือของเราวันนี้

เราจะใช้เครื่องมือมาตรฐานหลัก 3 ตัวในการทำ Workshop ครั้งนี้:

1. Windows CMD

Command Prompt สำหรับสั่งการระบบควบคุมเวอร์ชันผ่านตัวอักษรเพื่อความรวดเร็วและแม่นยำ

2. VS Code

Source Code Editor ชั้นนำเพื่อใช้แก้ไขข้อความ โค้ด และเรียกดูพรีวิวเอกสารแบบจัดฟอร์แมต

3. GitHub Cloud

เซิร์ฟเวอร์ปลายทางสำหรับการเผยแพร่ผลงานให้พร้อมถูกเข้าถึงได้จากทุกมุมโลก

ทำไมเราถึงเริ่มฝึกจาก Command Line (CMD)?

การเชื่อมต่อโปรแกรมผ่าน GUI หรือปุ่มกดอัตโนมัติอาจจะดูสะดวกสบายในช่วงแรก แต่ในวิชาชีพ Software Engineering การฝึกพิมพ์สั่งผ่าน CMD สำคัญมากด้วยเหตุผลดังนี้:

  • ความโปร่งใสในกระบวนการ: ช่วยให้เข้าใจ "ตรรกะและเบื้องหลัง" การทำงานของระบบ Git อย่างแท้จริง ไม่ถูกซ่อนโดยหน้าตา UI
  • การขยายระบบสู่อนาคต: เมื่อต้องทำงานกับ Cloud Server หรือการ Deploy ระบบอัตโนมัติ (CI/CD) จะไม่มีหน้าจอให้คลิกปุ่ม ต้องสั่งผ่าน Terminal ทั้งสิ้น

เฟสที่ 1

การเตรียมระบบในเครื่องคอมพิวเตอร์

เริ่มต้นตรวจสอบสิทธิ์และตั้งชื่อตัวตนกับโปรแกรม Git

Step 0: การเช็คความพร้อมของระบบ Git

0

พิมพ์คำสั่งนี้ลงในหน้าต่าง Command Prompt เพื่อตรวจสอบว่าเครื่องคอมพิวเตอร์จำคำสั่งของ Git ได้แล้วหรือยัง:

git --version

ผลลัพธ์ที่ได้ควรแสดงเลขเวอร์ชันของซอฟต์แวร์ออกมา เช่น:

git version 2.45.2.windows.1

หากพบข้อความแจ้งเตือนสีแดงว่า 'git' is not recognized... แปลว่าระบบ Path ของเครื่องมีปัญหา ให้ยกมือแจ้งอาจารย์ผู้ช่วยสอนทันที

Step 1: การประกาศตัวตนกับ Git Engine

1
ทำเพียงครั้งเดียวในเครื่องนี้

พิมพ์สองคำสั่งนี้เพื่อฝังชื่อและอีเมลลงไปในระบบประวัติการพิมพ์โค้ด (อย่าลืมเปลี่ยนเป็นข้อมูลจริงของคุณ):

git config --global user.name "Somsak Lanna"
git config --global user.email "somsak_l@rmutl.ac.th"

และกำหนดให้ชื่อกิ่งหลักเริ่มต้นเป็นมาตรฐานสากล:

git config --global init.defaultBranch main

Step 2: การสร้างโฟลเดอร์สำหรับ Portfolio

2

เราจะย้ายตำแหน่งการทำงานไปที่พื้นที่โฟลเดอร์ Documents และใช้คำสั่งสร้างโฟลเดอร์ขึ้นมาใหม่:

cd %USERPROFILE%\Documents
mkdir se-learning-portfolio
cd se-learning-portfolio

ทดสอบพิมพ์คำสั่งเช็คตำแหน่งปัจจุบัน เพื่อความชัวร์ว่าเราอยู่ในโฟลเดอร์ใหม่แล้ว:

cd

ผลลัพธ์บนจอควรเป็นเส้นทางลงท้ายด้วย: \Documents\se-learning-portfolio

Step 3: การเปิดโฟลเดอร์งานด้วย VS Code

3

พิมพ์คำสั่งลัดนี้ใน CMD เพื่อสั่งให้โปรแกรม Visual Studio Code เปิดโฟลเดอร์โปรเจกต์นี้ขึ้นมาทำงานทันที:

code .

จุดสังเกตคือมีเว้นวรรคหนึ่งช่องตามด้วยเครื่องหมายจุด (Dot) ซึ่งหมายถึง "โฟลเดอร์นี้ปัจจุบันที่เรายืนอยู่"

เฟสที่ 2

การเขียนเอกสารด้วยภาษา Markdown

ทำความรู้จักโครงสร้างเอกสารมาตรฐานและสร้างไฟล์ README.md เล่มแรก

ภาษา Markdown คืออะไร?

Markdown เป็นภาษาประเภท Lightweight Markup Language ที่ออกแบบมาให้นักพัฒนาซอฟต์แวร์เขียนเอกสารคู่มือประกอบโปรแกรมได้อย่างรวดเร็ว

"เขียนง่ายเหมือนพิมพ์ข้อความธรรมดา แต่แปรสภาพเป็นหน้าเว็บที่สวยงามและอ่านง่ายได้ทันที"

เป็นฟอร์แมตหลักที่ระบบหลังบ้านของ GitHub คอยอ่านเพื่อแปลงไปเป็นหน้าสารบัญคู่มือของทุกๆ โปรเจกต์ทั่วโลก

ตารางเรียนรู้ Syntax ภาษา Markdown (ตอนที่ 1)

เครื่องหมายที่เราพิมพ์ (Syntax) ความหมายและการทำงาน ตัวอย่างผลลัพธ์บนจอภาพ
# ชื่อหัวข้อ หัวข้อใหญ่ที่สุดลำดับที่ 1 (Heading 1)

ชื่อหัวข้อ
## หัวข้อย่อย หัวข้อย่อยลำดับที่ 2 (Heading 2)

หัวข้อย่อย
### หัวข้อย่อยระดับสาม หัวข้อย่อยลำดับที่ 3 (Heading 3)

หัวข้อย่อยระดับสาม

ตารางเรียนรู้ Syntax ภาษา Markdown (ตอนที่ 2)

เครื่องหมายที่เราพิมพ์ (Syntax) ความหมายและการทำงาน ตัวอย่างผลลัพธ์บนจอภาพ
**ข้อความตัวหนา** เน้นคำสำคัญหรือหัวข้อย่อยในย่อหน้า ข้อความตัวหนา
- รายการที่หนึ่ง สร้างรายการแบบจุดกลมนำหน้า (Unordered List) • รายการที่หนึ่ง
[ลิงก์เว็บ](https://...) ผูกข้อความเข้ากับลิงก์ URL ปลายทาง ลิงก์เว็บ

Step 4-5: สร้างเนื้อหาเอกสารแนะนำตัวเอง

# ข้อมูลการเรียนรู้ของฉัน

เฟสที่ 3

การจัดการและบันทึกข้อมูลด้วยคอมมานด์ไลน์ Git

เข้าสู่การเป็นวิศวกรซอฟต์แวร์อย่างเต็มตัวด้วยการหัดควบคุมสถานะของไฟล์

Step 6: การเบิกเนตรคลังเก็บไฟล์ (Git Init)

6

สลับหน้าจอกลับมาที่โปรแกรม CMD (ตรวจให้มั่นใจว่ายังอยู่ในโฟลเดอร์โปรเจกต์) แล้วสั่งเริ่มการทำงานของ Git:

git init

ระบบจะทำการแอบสร้างโฟลเดอร์ลับชื่อ .git ขึ้นมาเพื่อใช้ทำตัวเป็นกล้องวงจรปิดคอยเก็บข้อมูล

ผลลัพธ์บนจอ: Initialized empty Git repository in .../se-learning-portfolio/.git/

Step 7: คาถาเช็คสถานะระบบ (Git Status)

7

พิมพ์คำสั่งคอมมานด์ที่สำคัญและถูกใช้บ่อยที่สุดเพื่อตรวจส่องสถานะปัจจุบันของโฟลเดอร์:

git status

หน้าจอจะแสดงชื่อไฟล์ README.md ออกมาเป็น อักษรสีแดง ภายใต้หัวข้อ Untracked files

Untracked files:
  (use "git add <file>..." to include in what will be committed)
      README.md

ความหมายคือ: Git รู้แล้วว่ามีไฟล์นี้เกิดขึ้นใหม่ในพื้นที่ทำงาน แต่เรายังไม่ได้ออกคำสั่งให้ Git เริ่มจดจำหรือติดตามข้อมูลของไฟล์นี้

Step 8: การนำไฟล์วางบน Staging Area (Git Add)

8

เราต้องย้ายไฟล์ที่แก้ไขเข้าไปอยู่ใน "พื้นที่เตรียมตัว" หรือที่เรียกว่า Staging Area โดยใช้คำสั่ง:

git add README.md

หลังจากนั้น ลองเรียกใช้คำสั่งเช็คสถานะซ้ำอีกรอบทันที:

git status

ชื่อไฟล์ README.md จะเปลี่ยนร่างกลายเป็น อักษรสีเขียว แปลว่าไฟล์นี้ได้ไปนอนรอในตะกร้าเตรียมพร้อมที่จะถูกกดบันทึกประวัติถาวรในขั้นตอนต่อไปเรียบร้อยแล้ว

Step 9: การลั่นไกบันทึกประวัติลงเครื่อง (Git Commit)

9

ทำการบันทึกภาพ Snapshot ของสถานะโค้ดในเครื่องคอมพิวเตอร์อย่างถาวร พร้อมบันทึกข้อความสรุปงานสั้นๆ:

git commit -m "Create initial learning portfolio with README"

ลองใช้คำสั่งขอดูสมุดบันทึกประวัติศาสตร์ของโปรเจกต์เราดูว่าขึ้นข้อมูลหรือยัง:

git log --oneline

จะปรากฏรหัส Hash Code ของ Commit พร้อมข้อความโน้ตของคุณอย่างสง่างาม ถือเป็นอันเสร็จสิ้นกระบวนการบนเครื่อง Local

เฟสที่ 4

การเชื่อมโยงระบบสู่เครือข่ายอินเทอร์เน็ต GitHub

ผลักดันผลงานของคุณจากเครื่องคอมพิวเตอร์ขึ้นสู่ระบบคลาวด์สาธารณะ

Step 10: เปิดทำเลที่ตั้งบนระบบ GitHub Cloud

10
  1. เปิดเว็บเบราว์เซอร์ไปที่เว็บไซต์ GitHub.com
  2. กดปุ่มเครื่องหมายบวก (+) หรือปุ่ม New Repository
  3. Repository Name ตั้งชื่อว่า: se-learning-portfolio
  4. เลือกสถานะเป็น Public เพื่อเปิดให้ทุกคนส่องผลงานได้
  5. ห้ามติ๊กเลือกเด็ดขาด ตรงช่อง Add a README file
  6. กดปุ่มสีเขียว Create repository

Step 11: การผูกสายสัญญานโยงคอมเข้าหาเว็บ

11

คัดลอกลิงก์ที่อยู่ URL (แบบ HTTPS) จากหน้าเว็บโปรเจกต์ของ GitHub นำมาพิมพ์สั่งใน CMD เพื่อบอกให้เครื่องของเรารู้จักที่อยู่ปลายทางบนอินเทอร์เน็ต:

git remote add origin https://github.com/เปลี่ยนเป็นชื่อUSERของคุณ/se-learning-portfolio.git

เพื่อความมั่นใจว่าไม่มีอะไรผิดพลาด ลองพิมพ์คำสั่งตรวจสอบรายชื่อ Remote Server ปลายทาง:

git remote -v

จอภาพจะต้องแสดงลิงก์ URL ของคลังไฟล์ GitHub ของคุณขึ้นมาทั้งในวงเล็บ (fetch) และ (push)

Step 12: การผลักข้อมูลประวัติพุ่งขึ้นสู่ Cloud (Push)

12

ยิงคอมมานด์คำสั่งเพื่อดันข้อมูลประวัติทั้งหมดที่บันทึกไว้ในเครื่องคอมพิวเตอร์ให้เดินทางขึ้นสู่ GitHub Cloud เซิร์ฟเวอร์:

git push -u origin main

ระบบยืนยันสิทธิ์ความปลอดภัย: จะมีหน้าต่าง Pop-up ของระบบ Windows เด้งขึ้นมาขอสิทธิ์ล็อกอิน ให้คุณทำการคลิกปุ่มเลือก Sign in via Browser จากนั้นกดยืนยันในหน้าเว็บเบราว์เซอร์ให้เรียบร้อย ข้อมูลก็จะเริ่มอัปโหลดทันทีครับ

วงจรกระบวนงานประจำวันของ Software Engineer

หลังจากเซ็ตระบบครั้งแรกผ่านไปแล้ว เมื่อคุณมีการกลับมาแก้ไขเพิ่มข้อมูลสกิลในไฟล์เอกสาร README.md ในวันต่อๆ ไป วงจรชีวิตที่คุณต้องทำซ้ำในหน้าต่าง CMD มีเพียงแค่คาถา 3 บรรทัดนี้เท่านั้นครับ:

# 1. นำชิ้นงานที่แก้ไขแล้วใส่กล่องพัสดุ
git add README.md

# 2. ปิดกล่อง ประทับตราประวัติพร้อมเขียนโน้ตกำกับ
git commit -m "Update my current skills database"

# 3. ส่งขนส่งพัสดุพุ่งขึ้นไปหาเซิร์ฟเวอร์คลาวด์
git push

ตารางวิเคราะห์และแก้ไขปัญหาเมื่อระบบพัง

1. 'git' ไม่ใช่คำสั่งที่รู้จัก

สาเหตุ: คอมพิวเตอร์ไม่ได้ติดตั้งโปรแกรม Git หรือยังไม่ได้รีสตาร์ทหน้าต่าง CMD หลังติดตั้งเสร็จ
วิธีแก้: ปิดคอมมานด์พรอมต์แล้วเปิดใหม่ หรือลงตัวโปรแกรมซ้ำ

2. Push Error (Rejected)

สาเหตุ: เกิดจากเราเผลอไปกดติ๊กสร้างไฟล์ README บนหน้าเว็บ GitHub ล่วงหน้า ทำให้ประวัติสองฝั่งขัดแย้งกัน
วิธีแก้: แนะนำให้ปรึกษาอาจารย์เพื่อเคลียร์ระบบประวัติศาสตร์ใหม่

3. ข้อมูล Remote ผิดพลาด

สาเหตุ: วางคัดลอกคีย์ URL ของ GitHub ผิดคลังโปรเจกต์
วิธีแก้: ใช้คำสั่งล้างข้อมูลเก่า git remote remove origin แล้วก๊อปปี้คีย์มาวางเชื่อมต่อใหม่อีกครั้ง

คำถามทบทวนตรรกะลำดับกระบวนการทำงาน

จงพิจารณาการเรียงลำดับคอมมานด์คำสั่งในการส่งงานขึ้นระบบครั้งแรก ข้อใดเรียงลำดับลอจิกขั้นตอนได้ถูกต้อง?

A. push
B. init
C. commit
D. add

ลำดับที่ถูกต้องคือ: B (init) → D (add) → C (commit) → A (push)

เราต้องสร้างคลังคอยติดตามประวัติก่อน จากนั้นรวบรวมไฟล์ใส่กล่อง สั่งบันทึกภาพลงเครื่อง แล้วจึงส่งขึ้นอินเทอร์เน็ตครับ

แนวคิด Learning Portfolio ของหลักสูตรวิศวกรรมซอฟต์แวร์

การเรียนวิชาชีพวิศวกรรมคอมพิวเตอร์และซอฟต์แวร์ในยุคปัจจุบัน คะแนนสอบทฤษฎีในห้องเรียนอาจไม่ใช่คำตอบสุดท้ายของการได้งานทำ

"คลังผลงานบน GitHub ของคุณจะทำหน้าที่เป็นพยานบุคคลที่โปร่งใสที่สุดในการยืนยันความสามารถต่อผู้ประกอบการ"

โปรเจกต์ Lab ทุกชิ้น, มินิโปรเจกต์ทุกวิชา จะถูกนำมาจัดวางสะสมไว้ในพื้นที่ Repo นี้อย่างมีระเบียบเพื่อให้พร้อมใช้งานในการยื่นฝึกงานในชั้นปีที่ 3

กิจกรรมท้ายชั่วโมง: สลับกันรีวิว (Peer Review)

ให้นักศึกษาจับคู่กับเพื่อนที่นั่งอยู่ด้านข้าง สลับลิงก์ URL คลังผลงาน GitHub ของตนเองให้เพื่อนช่วยเปิดตรวจสอบผ่านเครื่องคอมพิวเตอร์:

  • หน้าเว็บ GitHub แสดงผลไฟล์ README.md ครบถ้วนชัดเจนหรือไม่?
  • มีการสะกดคำผิดในจุดใดหรือเปล่า?
  • ให้คำชมเชยชื่นชมเพื่อน และคำแนะนำในการปรับปรุงแก้ไข 1 ข้อ
