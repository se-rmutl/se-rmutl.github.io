กรณีศึกษา: The Million Dollar Mistake

บทเรียนราคาแพงจากความผิดพลาดในการกำหนดความต้องการ คือสิ่งที่ย้ำเตือนถึงความสำคัญของบทเรียนในวันนี้

จรวด Ariane 5 ระเบิด (1996)

หลังปล่อยตัวเพียง 37 วินาที จรวดมูลค่ากว่า 500 ล้านดอลลาร์ได้ระเบิดกลางอากาศ สาเหตุหลักมาจากการแปลงข้อมูลผิดพลาด แต่รากของปัญหาที่แท้จริงคือ...

ปัญหาในเอกสาร Requirements: ไม่ได้ระบุข้อจำกัดของระบบย่อยอย่างชัดเจน และไม่มีการตรวจสอบความต้องการข้ามระบบ

Active Learning Question

หากคุณเป็น Requirements Engineer ในโครงการนี้ คุณจะเขียน Requirement เพื่อป้องกันปัญหานี้ได้อย่างไร?

แนวคำตอบ:

"ระบบ Inertial Reference System (IRS) จะต้องสามารถรองรับค่าความเร็วแนวนอน (horizontal velocity) ที่มีค่าสูงสุดเท่ากับค่าที่คาดการณ์ไว้ใน trajectory ของ Ariane 5 ได้โดยไม่มี overflow error เกิดขึ้น และต้องมี exception handling สำหรับกรณีที่ค่าเกินกว่าที่ระบบจะรับได้"

หัวใจของการสื่อสาร: Software Requirements Specification (SRS)

SRS คือเอกสารที่เป็นทางการซึ่งทำหน้าที่เป็นพิมพ์เขียวของซอฟต์แวร์ ส่วนนี้จะพาคุณไปทำความรู้จักกับโครงสร้างมาตรฐานและวิธีตรวจสอบคุณภาพของมัน

SRS คืออะไร?

เอกสารที่อธิบายอย่างละเอียดว่าซอฟต์แวร์ควรทำอะไรและทำงานภายใต้เงื่อนไขใด เป็นหัวใจสำคัญในการสื่อสารระหว่าง stakeholders, กำหนดขอบเขตโครงการ และเป็นพื้นฐานของการออกแบบและทดสอบ โครงการที่มี SRS ที่ดีมีโอกาสสำเร็จสูงกว่าและช่วยลดต้นทุนการแก้ไขได้อย่างมหาศาล

โครงสร้างหลักตามมาตรฐาน IEEE 830

ลองตรวจสอบคุณภาพ Requirements

คลิกที่การ์ดเพื่อดูปัญหาและแนวทางแก้ไข

"The system shall be user-friendly and provide good performance."

❌ Ambiguous & Not Verifiable

แก้ไข: "The system shall respond within 2s under normal load (≤100 users)."

"When user clicks login, system validates credentials, if valid, redirect to dashboard, otherwise show error..."

❌ Not Atomic (ทำหลายอย่าง)

แก้ไข: แยกเป็นหลาย requirements ย่อยสำหรับแต่ละ action

"The system must process transactions quickly."

❌ Ambiguous & Not Verifiable

แก้ไข: "System must process 100 transactions per second."

ศิลปะการเขียน Requirement: จากแนวคิดสู่ข้อความที่ชัดเจน

การเขียน Requirement ที่ดีคือทักษะสำคัญ เราจะมาเรียนรู้เครื่องมือและเทคนิคที่จะช่วยให้คุณสื่อสารความต้องการได้อย่างแม่นยำ ไม่ว่าจะเป็นในรูปแบบของ Waterfall หรือ Agile

Use Case: เล่าเรื่องของระบบ

อธิบายปฏิสัมพันธ์ระหว่าง Actor กับระบบเพื่อบรรลุเป้าหมาย เน้นขั้นตอนการทำงาน (How) และแสดงภาพรวมด้วย Use Case Diagram

User Story: เสียงจากผู้ใช้

ประโยคสั้นๆ ที่บอกความต้องการของผู้ใช้ เน้นคุณค่าที่ผู้ใช้จะได้รับ (Why) เหมาะกับ Agile Development

As a [type of user],

I want [to perform some task]

So that [I can achieve some goal].

Acceptance Criteria: Definition of Done

เงื่อนไขที่ระบุอย่างชัดเจนว่า User Story จะถือว่า "เสร็จ" เมื่อไหร่

รูปแบบ Given-When-Then:

Given I am adding a new expense

When I select a category from the dropdown

Then the expense should be saved with the selected category

การตรวจสอบความถูกต้อง: Verification & Validation

การเขียน Requirement ที่ดีนั้นยังไม่พอ เราต้องแน่ใจว่าสิ่งที่เราเขียนนั้นถูกต้อง (Verification) และเป็นสิ่งที่ผู้ใช้ต้องการจริงๆ (Validation)

Verification

"Are we building the product right?"

ตรวจสอบว่าทำตาม specification ถูกต้องหรือไม่ (Process-oriented)

  • Requirements Reviews
  • Inspections
  • Automated Analysis

Validation

"Are we building the right product?"

ตรวจสอบว่าผลิตภัณฑ์ตอบโจทย์ผู้ใช้ได้จริงหรือไม่ (User-oriented)

  • Prototyping
  • User Acceptance Testing (UAT)
  • Stakeholder Reviews

การประกันคุณภาพและการจัดการเอกสาร

เรียนรู้กระบวนการและเครื่องมือที่จะช่วยรักษาคุณภาพของ Requirements ตลอดทั้งโครงการ ตั้งแต่การเขียน, การตรวจสอบ, ไปจนถึงการจัดการเมื่อมีการเปลี่ยนแปลง

แนวปฏิบัติที่ดีที่สุดในการเขียนเอกสาร

  • Use Active Voice: "The system shall encrypt data" (ดีกว่า "Data will be encrypted")
  • Avoid Weak Words: ใช้ "shall", "must" แทน "should", "might"
  • Be Specific with Numbers: "Support 1000 concurrent users" (ดีกว่า "handle many users")
  • Use Consistent Terminology: สร้าง Glossary เพื่อกำหนดคำศัพท์ที่ใช้ตรงกัน

เช็คลิสต์ประกันคุณภาพ SRS

Traceability Matrix: เชื่อมโยงทุกอย่างเข้าด้วยกัน

ตารางที่ช่วยติดตามความสัมพันธ์ระหว่าง Business Need, Requirement, Design, และ Test Case เพื่อวิเคราะห์ผลกระทบเมื่อมีการเปลี่ยนแปลง

กระบวนการจัดการการเปลี่ยนแปลง (Change Management)

เมื่อความต้องการเปลี่ยน ต้องมีกระบวนการที่เป็นระบบเพื่อควบคุมผลกระทบ

  1. Change Request Submission: ผู้ร้องขอยื่นฟอร์มพร้อมเหตุผล
  2. Impact Analysis: ทีมวิเคราะห์ผลกระทบต่อเวลา, งบประมาณ, และเทคนิค
  3. Change Review Board: คณะกรรมการ (Product Owner, Tech Lead, PM) ตัดสินใจ
  4. Decision & Implementation: อนุมัติ, เลื่อน, หรือปฏิเสธการเปลี่ยนแปลง
