Outline

🤔 UML คืออะไร? (What is UML?)

ENGSE206

UML: Unified Modeling Language

  • 🔵เป็น "ภาษารูปภาพ" (Visual Language) มาตรฐานที่ใช้ในการ สร้างแบบจำลอง (Modeling) ระบบซอฟต์แวร์
  • 📜ไม่ได้เป็นภาษาโปรแกรมมิ่ง แต่เป็น พิมพ์เขียว (Blueprint) ที่ช่วยให้ทุกคนในทีมเห็นภาพเดียวกัน
  • 🤝ช่วยในการ สื่อสาร ระหว่าง Business Analyst, Software Architect, Developer, และ Tester

ทำไมเราต้องใช้ UML?

ลดความกำกวม (Reduce Ambiguity)

ภาพหนึ่งภาพแทนคำพูดนับพันคำ ทำให้เข้าใจตรงกันได้ง่ายกว่าการใช้เพียงข้อความ

เห็นภาพรวม (Visualize System)

ช่วยให้เข้าใจโครงสร้างและความสัมพันธ์ที่ซับซ้อนของระบบทั้งหมดได้ในหน้าเดียว

เป็นเอกสาร (Documentation)

ใช้เป็นเอกสารประกอบการออกแบบที่ชัดเจน เป็นมาตรฐาน และง่ายต่อการส่งมอบงาน

วิเคราะห์และออกแบบ (Analysis & Design)

เป็นเครื่องมือสำคัญในการค้นหาปัญหาที่อาจเกิดขึ้นในการออกแบบและหาโซลูชันก่อนลงมือเขียนโค้ด

Introduction to UML

🧩 ประเภทของ UML Diagrams

ENGSE206

1. Structural Diagrams (แผนภาพโครงสร้าง)

แสดงโครงสร้าง "สถิต" (Static) ของระบบ หรือสิ่งต่างๆ ที่ประกอบกันเป็นระบบ ณ เวลาใดเวลาหนึ่ง ไม่สนใจการเปลี่ยนแปลงตามเวลา เปรียบเสมือนการดูพิมพ์เขียวของอาคาร ที่เห็นว่ามีห้องอะไรบ้าง แต่ไม่เห็นว่าคนเดินไปมาอย่างไร

ตัวอย่างที่เราจะเรียน:

  • Class Diagram
  • Component Diagram
  • Deployment Diagram

2. Behavioral Diagrams (แผนภาพพฤติกรรม)

แสดงพฤติกรรม "พลวัต" (Dynamic) ของระบบ หรือการทำงาน, การเปลี่ยนแปลง, และปฏิสัมพันธ์ระหว่างส่วนต่างๆ เมื่อเวลาผ่านไป เปรียบเสมือนการดูกล้องวงจรปิดในอาคาร ที่เห็นกิจกรรมและการเคลื่อนไหวต่างๆ ที่เกิดขึ้น

ตัวอย่างที่เราจะเรียน:

  • Use Case Diagram
  • Activity Diagram
  • Sequence Diagram
Structural vs. Behavioral

🔗 ความสัมพันธ์ระหว่าง UML และ C4 Model

ENGSE206

C4 Model ช่วยให้เรา "ซูม" เข้าไปในสถาปัตยกรรมซอฟต์แวร์ใน 4 ระดับ ส่วน UML คือเครื่องมือที่เราใช้ "วาด" ภาพในแต่ละระดับการซูมนั้น

C4 Level มุมมอง (View) UML Diagram ที่เหมาะสมที่สุด เหตุผล
C1: System Context ภาพรวมที่สุด Use Case Diagram แสดงว่า ใคร ใช้งานระบบ และระบบทำ อะไร ได้บ้าง
C2: Containers เทคโนโลยีหลัก Deployment / Component Diagram แสดง Container หลักๆ (Web App, API, DB) และเทคโนโลยีที่ใช้
C3: Components ส่วนประกอบย่อย Component / Class / Sequence Diagram แสดง Component ภายใน Container และการสื่อสารระหว่างกัน
C4: Code รายละเอียดโค้ด Class / Sequence Diagram แสดงโครงสร้าง Class และลำดับการเรียก Method

เราอาจใช้ Deployment Diagram เพื่อแสดงว่า Web App, API Server, และ Database ถูก deploy ที่ไหน

C3 - Components:

เราใช้ Component Diagram เพื่อแยกส่วนประกอบใน API Server และใช้ Sequence Diagram เพื่อแสดงว่า Component เหล่านี้คุยกันอย่างไร

C4 - Code:

เราใช้ Class Diagram เพื่อออกแบบโครงสร้างของโค้ดในแต่ละ Component

Applying UML to C4 Levels

1. Use Case Diagram

ENGSE206

คืออะไร?

แผนภาพที่แสดง มุมมองจากผู้ใช้งาน (User's Perspective) เพื่ออธิบายว่า "ใคร (Actor)" ทำ "อะไร (Use Case)" กับระบบบ้าง มันคือจุดเริ่มต้นที่ดีที่สุดในการทำความเข้าใจขอบเขต (Scope) และความต้องการหลัก (Main Requirements) ของระบบ

เหมาะกับ C4 Level ไหน?

C1 (System Context) และ C2 (Containers)

เหตุผล: ช่วยระบุ Actors (ผู้ใช้และระบบภายนอก) และฟังก์ชันหลักของระบบได้อย่างชัดเจน ซึ่งเป็นหัวใจของ C1 และช่วยให้เห็นปฏิสัมพันธ์หลักระหว่างผู้ใช้กับ Containers ในระดับ C2

ตอบคำถามอะไร?

"ระบบของเราให้บริการอะไรกับใครบ้าง?"

เหมาะสำหรับ

การกำหนดขอบเขตโครงการ, การสื่อสารกับ Stakeholders ที่ไม่ใช่สายเทคนิค, และการระบุ Functional Requirements หลัก

ข้อควรระวัง

อย่าลงรายละเอียดมากเกินไป (เช่น UI design) และแผนภาพนี้ไม่ได้บอก "ลำดับ" การทำงาน

2. Activity Diagram

ENGSE206

คืออะไร?

แผนภาพที่แสดง Workflow หรือ ลำดับขั้นตอนการทำงาน ของระบบแบบละเอียด เหมาะสำหรับการอธิบาย Logic ที่ซับซ้อนภายใน Use Case หรือ Component หนึ่งๆ

ดีกว่า Flowchart อย่างไร?

Activity Diagram มีสัญลักษณ์เฉพาะสำหรับงานที่ทำพร้อมกัน (Parallel Activities) และการแบ่งหน้าที่ความรับผิดชอบ (Swimlanes) ซึ่ง Flowchart ทั่วไปไม่มี

เหมาะกับ C4 Level ไหน?

C3 (Components)

เหตุผล: ช่วยอธิบายรายละเอียดขั้นตอนการทำงานที่เกิดขึ้นภายใน Component ได้อย่างละเอียด เช่น Flow การเปลี่ยนสถานะของ Agent ที่อาจเกี่ยวข้องกับหลายส่วน

ตอบคำถามอะไร?

"ขั้นตอนการทำงานของฟีเจอร์นี้เป็นอย่างไรตั้งแต่ต้นจนจบ?"

เหมาะสำหรับ

การอธิบาย Business Process, การออกแบบ Logic ที่ซับซ้อน, และการแสดงให้เห็นว่าส่วนต่างๆ ทำงานร่วมกันอย่างไรใน Workflow

ข้อควรระวัง

อาจจะซับซ้อนเกินไปสำหรับฟังก์ชันง่ายๆ และเน้นที่ Flow การควบคุม ไม่ใช่ Flow ของข้อมูล

3. Class Diagram

ENGSE206

คืออะไร?

แผนภาพที่แสดง โครงสร้างของ Class ในระบบ เปรียบเสมือนพิมพ์เขียวของโค้ด (Code Blueprint) ที่แสดงองค์ประกอบหลักของระบบเชิงวัตถุ (Object-Oriented) ประกอบด้วย Attributes (ข้อมูล), Methods (พฤติกรรม), และ Relationships ระหว่าง Class

เหมาะกับ C4 Level ไหน?

C3 (Components) และ C4 (Code)

เหตุผล: ในระดับ C3 ใช้เพื่อแสดง Class หลักๆ ที่ประกอบกันเป็น Component ส่วนในระดับ C4 ใช้เพื่อลงรายละเอียดโครงสร้างโค้ดทั้งหมดในระดับที่ Developer สามารถนำไป implement ได้ทันที

ตอบคำถามอะไร?

"ส่วนประกอบต่างๆ ของโค้ดเรามีอะไรบ้าง และมันเชื่อมโยงกันอย่างไร?"

เหมาะสำหรับ

การออกแบบโครงสร้างโค้ด, การทำความเข้าใจโค้ดเดิม (Reverse Engineering), และการสื่อสารโครงสร้างระหว่างนักพัฒนา

ข้อควรระวัง

อาจจะละเอียดและซับซ้อนเกินไปสำหรับคนที่ไม่ใช่ Technical และต้องคอยอัปเดตให้ตรงกับโค้ดจริงเสมอ

3. Class Diagram - สัญลักษณ์

ENGSE206

Class: กล่อง 3 ส่วน (Name, Attributes, Methods)

Visibility: `+` Public, `-` Private, `#` Protected

สัญลักษณ์ ความสัมพันธ์ (Relationship) ความหมาย
─── Association "มีความสัมพันธ์กับ" (เช่น Supervisor manages Agent)
◇── Aggregation "มีส่วนประกอบเป็น" (has-a), แต่ส่วนประกอบอยู่ได้ด้วยตัวเอง (เช่น Team has Agents)
◆── Composition "เป็นเจ้าของ" (owns-a), ส่วนประกอบอยู่ด้วยตัวเองไม่ได้ (เช่น Car owns an Engine)
△── Generalization (Inheritance) "เป็นชนิดหนึ่งของ" (is-a), การสืบทอดคุณสมบัติ (เช่น Supervisor is a User)

4. Sequence Diagram

ENGSE206

คืออะไร?

  • แผนภาพที่แสดง ปฏิสัมพันธ์ (Interaction) ระหว่าง Object หรือ Component ตามลำดับเวลา
  • เน้นการแสดง "ลำดับ" ของการส่งข้อความ (Message) หรือการเรียกใช้ Method ระหว่างส่วนต่างๆ ของระบบ
  • ช่วยให้เห็นภาพว่า Workflow หนึ่งๆ ทำงานอย่างไรตั้งแต่ต้นจนจบ

เหมาะกับ C4 Level ไหน?

C2 (Containers) และ C3 (Components)

เหตุผล: ระดับ C2 แสดงลำดับการสื่อสารระหว่าง Containers (Web App → API → DB), ส่วนระดับ C3 แสดงลำดับการเรียกใช้ Method ระหว่าง Components ภายใน Container เดียวกัน

ตอบคำถามอะไร?

"เมื่อเกิดเหตุการณ์ X, ส่วนต่างๆ ของระบบคุยกันอย่างไรและเรียงลำดับอย่างไร?"

เหมาะสำหรับ

การทำความเข้าใจ Runtime Behavior, การออกแบบ API ที่ซับซ้อน, และการหาจุดที่เกิดความล่าช้า (Bottleneck)

ข้อควรระวัง

การใส่ Logic ที่ซับซ้อน (เช่น loops, conditions) มากเกินไปจะทำให้อ่านยาก ควรใช้ Activity Diagram แทน

5. Component Diagram

ENGSE206

คืออะไร?

แผนภาพที่แสดง โครงสร้างของส่วนประกอบ (Components) ของระบบ และความสัมพันธ์ระหว่างกัน Component คือส่วนของซอฟต์แวร์ที่สามารถทำงานแยกกันได้ และมี Interface สำหรับการติดต่อกับภายนอก เช่น Library, Framework, หรือกลุ่มของ Class ที่ทำงานร่วมกัน ช่วยให้เห็นภาพการแบ่ง Module ของระบบ

เหมาะกับ C4 Level ไหน?

C3 (Components)

เหตุผล: ชื่อก็บอกอยู่แล้ว! แผนภาพนี้สร้างขึ้นเพื่อแสดง Components ที่ประกอบกันเป็น Container หนึ่งๆ โดยเฉพาะ ซึ่งตรงกับเป้าหมายของ C3 ทุกประการ

ตอบคำถามอะไร?

"ระบบของเราประกอบด้วยโมดูลอะไรบ้าง และแต่ละโมดูลพึ่งพากันอย่างไร?"

เหมาะสำหรับ

การออกแบบสถาปัตยกรรมระดับกลาง, การแบ่งงานให้ทีม, และการทำความเข้าใจ Dependencies ระหว่างส่วนต่างๆ

ข้อควรระวัง

เป็นแผนภาพเชิงแนวคิด (Conceptual) ไม่ได้แสดงรายละเอียดระดับ Class หรือ Object

6. Deployment Diagram

ENGSE206

คืออะไร?

แผนภาพที่แสดง โครงสร้างทางกายภาพ (Physical Architecture) ของระบบ อธิบายว่าส่วนประกอบซอฟต์แวร์ (Artifacts เช่น .jar, .exe) ถูกนำไปติดตั้ง (Deploy) บนฮาร์ดแวร์ (Nodes เช่น Server) อย่างไร และแสดงการเชื่อมต่อทางเครือข่ายระหว่างฮาร์ดแวร์เหล่านั้น

เหมาะกับ C4 Level ไหน?

C2 (Containers)

เหตุผล: ช่วยให้เห็นภาพว่า Containers ต่างๆ (ซึ่งเป็น Artifacts) ถูก deploy บนโครงสร้างพื้นฐาน (Nodes) อย่างไร และสื่อสารกันผ่าน Protocol อะไร ซึ่งเป็นการเสริมมุมมองของ C2 ให้สมบูรณ์ยิ่งขึ้น

ตอบคำถามอะไร?

"ซอฟต์แวร์ของเราจะถูกนำไปติดตั้งบนเครื่องเซิร์ฟเวอร์และเชื่อมต่อกันอย่างไร?"

เหมาะสำหรับ

การวางแผน Infrastructure, การสื่อสารกับทีม System Admin/DevOps, และการทำความเข้าใจสภาพแวดล้อมของ Production

ข้อควรระวัง

ไม่ต้องแสดงทุก Node ที่มีในระบบ แต่เน้นเฉพาะส่วนที่สำคัญต่อสถาปัตยกรรมซอฟต์แวร์

🔄 Workflow การใช้ UML Diagrams

ENGSE206

Phase 1: Requirements Analysis

  1. Use Case Diagram - ระบุ Actors และฟังก์ชันหลัก
  2. Activity Diagram - วิเคราะห์ Business Process
  3. Validation - ตรวจสอบความถูกต้องกับ Stakeholders

Phase 2: System Design

  1. Component Diagram - ออกแบบ Architecture ระดับสูง
  2. Class Diagram - ออกแบบ Domain model

Phase 3: Detailed Design

  1. Sequence Diagram - ออกแบบ Method interactions
  2. Refined Class Diagram - เพิ่มรายละเอียด Methods และ Attributes
  3. Component Interfaces - ระบุ APIs และ Contracts

Phase 4: Implementation & Deployment

  1. Deployment Diagram - วางแผนการติดตั้งระบบจริง
  2. Testing - ใช้ Sequence diagrams สำหรับ Test scenarios

🚀 Best Practices สำหรับการใช้ UML

ENGSE206

1. เลือก Diagram ที่เหมาะสม

  • ไม่ต้องใช้ทุก Diagram ในทุกโครงการ
  • เลือกตาม complexity และ requirements ของโครงการ
  • มุ่งเน้น value ที่จะได้รับ ไม่ใช่การวาด Diagram ให้สวย

2. Keep It Simple

  • เริ่มจาก high-level แล้วค่อยลงรายละเอียด
  • หลีกเลี่ยงการใส่รายละเอียดมากเกินไปในภาพเดียว
  • ใช้ abstraction ให้เหมาะสมกับผู้ดู

3. Maintain Consistency

  • ใช้ naming conventions เดียวกันทั้งโครงการ
  • ใช้สีและรูปแบบให้สอดคล้องกัน
  • Update diagrams เมื่อมีการเปลี่ยนแปลง requirements

4. Focus on Communication

  • Diagram ต้องช่วยให้ทีมเข้าใจระบบได้ง่ายขึ้น
  • เพิ่ม notes และ descriptions เมื่อจำเป็น
  • ทำ diagram review กับทีมอย่างสม่ำเสมอ
