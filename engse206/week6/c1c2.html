Call Center Agents

🖼️ C2 - Presentation Tier (Frontend)

ส่วนติดต่อผู้ใช้งานสำหรับแต่ละกลุ่ม

🖥️

Agent Desktop App

เทคโนโลยี: Electron.js
ผู้ใช้: Agents
หน้าที่: จัดการสถานะ, รับข้อความ, Login/Logout ผ่านแอปบนเครื่องคอมพิวเตอร์

🌐

Supervisor Dashboard

เทคโนโลยี: React.js
ผู้ใช้: Supervisors
หน้าที่: ติดตามสถานะทีม Real-time, ส่งข้อความ, ดูภาพรวมทีม

📈

Management Portal

เทคโนโลยี: React.js
ผู้ใช้: Managers
หน้าที่: ดูรายงานวิเคราะห์, KPI dashboards, และข้อมูลเชิงลึก

🛠️

Admin Console

เทคโนโลยี: React.js
ผู้ใช้: Administrators
หน้าที่: จัดการผู้ใช้, ตั้งค่าระบบ, ดู Audit Logs

💡 Key Decision: ทำไมแยก Frontend เป็น 4 Containers? เพื่อให้แต่ละ Role มี Interface ที่เหมาะสม, ปลอดภัย, และดูแลรักษาง่าย

⚙️ C2 - Application Tier (Backend)

ส่วนประมวลผลกลางของระบบ

🔗

API Server

เทคโนโลยี: Node.js + Express
หน้าที่: เป็น RESTful API สำหรับจัดการข้อมูล, Business Logic, และ Validation

⚡

WebSocket Server

เทคโนโลยี: Socket.io
หน้าที่: จัดการการสื่อสารสองทางแบบ Real-time, Broadcast สถานะ, และส่งข้อความ

🔑

Authentication Service

เทคโนโลยี: Node.js + JWT
หน้าที่: จัดการ Login/Logout, สร้างและตรวจสอบ Token, และควบคุมสิทธิ์การเข้าถึง (RBAC)

🔄

Background Services

เทคโนโลยี: Node.js Workers
หน้าที่: ประมวลผลงานเบื้องหลัง เช่น สร้างรายงาน, รวบรวมข้อมูล, และส่งแจ้งเตือน

💡 Key Decision: ทำไมแยก API กับ WebSocket? เพื่อให้แต่ละส่วนขยายระบบ (Scale) ได้อย่างอิสระตามภาระงาน และป้องกันไม่ให้ส่วนหนึ่งล่มแล้วกระทบอีกส่วน (Fault Isolation)

💾 C2 - Data Tier (Database)

ส่วนจัดเก็บข้อมูลของระบบ

🗃️

MSSQL Database

เทคโนโลยี: Microsoft SQL Server
ข้อมูลที่เก็บ:

  • ข้อมูลหลัก (Master Data) เช่น บัญชีผู้ใช้, ทีม, การตั้งค่าระบบ
  • ข้อมูลที่ต้องการความถูกต้องสูง (ACID Compliance)
  • สถานะปัจจุบันของ Agent (OnlineAgents Table)

📄

MongoDB Database

เทคโนโลยี: MongoDB (NoSQL)
ข้อมูลที่เก็บ:

  • ข้อมูลที่เกิดขึ้นบ่อยและมีปริมาณมาก (High Volume)
  • ประวัติการเปลี่ยนสถานะ (StatusLogs)
  • ประวัติข้อความ (Message History)
  • ข้อมูลสำหรับ Analytics ที่ต้องการความยืดหยุ่น

💡 Key Decision: ทำไมใช้ Hybrid Database (MSSQL + MongoDB)? เพื่อเลือกใช้ฐานข้อมูลที่เหมาะสมกับลักษณะของข้อมูลแต่ละประเภท MSSQL สำหรับข้อมูลที่มีโครงสร้างชัดเจนและต้องการความสัมพันธ์ที่ซับซ้อน และ MongoDB สำหรับข้อมูลปริมาณมากที่ต้องการความเร็วในการเขียนและอ่าน

🔄 C2 - Communication & Protocols

ส่วนต่างๆ ของระบบสื่อสารกันอย่างไร?

📡

Frontend ↔ Backend

  • REST API (HTTPS): สำหรับการร้องขอข้อมูลทั่วไป (CRUD)
  • WebSocket (WSS): สำหรับการสื่อสาร Real-time สองทิศทาง
  • Authentication: ใช้ JWT Bearer Token ใน Header
⚙️

Internal Services

  • API ↔ WebSocket: เรียกใช้กันผ่าน Internal API Calls
  • API ↔ Background: ใช้ Job Queue (เช่น Redis) เพื่อมอบหมายงานแบบ Asynchronous
🔀

Database Connections

  • Connection Pooling: เพื่อจัดการการเชื่อมต่อฐานข้อมูลอย่างมีประสิทธิภาพ
  • Transaction: เพื่อรับประกันความถูกต้องของข้อมูลสำคัญ (ACID)

📈 C2 - Scalability & Performance

การออกแบบเพื่อรองรับการขยายตัวและประสิทธิภาพ

Horizontal Scaling Strategy

ระบบถูกออกแบบมาให้สามารถเพิ่มจำนวนเครื่อง Server (Instance) ในแต่ละส่วนเพื่อรองรับผู้ใช้ที่มากขึ้นได้:

  • Frontend: เพิ่ม Instance หลัง Load Balancer
  • API Server: เพิ่ม Instance ตามการใช้งาน CPU
  • WebSocket Server: ใช้ Redis Adapter เพื่อสื่อสารข้าม Instance
  • Database: ใช้ Read Replicas (MongoDB) และ Always On (MSSQL)

Performance Optimization

เทคนิคที่ใช้เพื่อเพิ่มความเร็วในการทำงาน:

  • Caching: ใช้ Redis เก็บข้อมูลที่เรียกใช้บ่อย
  • Database Indexing: สร้าง Index เพื่อเร่งความเร็วในการค้นหาข้อมูล Real-time
  • Connection Pooling: ลด Overhead ในการเชื่อมต่อฐานข้อมูล
  • CDN: ใช้ Content Delivery Network สำหรับไฟล์ Static

🔒 C2 - Security Architecture

การออกแบบด้านความปลอดภัย

Authentication Flow

  1. ผู้ใช้ Login เข้าสู่ระบบ
  2. Authentication Service ตรวจสอบและสร้าง JWT Token
  3. Frontend จัดเก็บ Token อย่างปลอดภัย
  4. ทุก Request ไปยัง API จะต้องแนบ JWT Token ไปใน Header
  5. มีระบบ Token Refresh เพื่อจัดการ Session

Network & Data Security

  • HTTPS/WSS: เข้ารหัสการสื่อสารทั้งหมดด้วย SSL/TLS
  • Internal Network: Service ภายในสื่อสารกันผ่าน Private Network
  • Password Hashing: จัดเก็บรหัสผ่านด้วยการ Hash (เช่น bcrypt)
  • Audit Logging: บันทึกเหตุการณ์สำคัญด้านความปลอดภัย
ระบบความปลอดภัยครอบคลุมตั้งแต่การยืนยันตัวตน, การเข้ารหัสข้อมูล, ไปจนถึงการป้องกันระดับเครือข่าย

ทำไมต้องรู้จัก Design Patterns ก่อนไป C3?

Design Patterns คือ "ท่าไม้ตาย" หรือ "พิมพ์เขียว" ที่ผ่านการพิสูจน์แล้วสำหรับการแก้ปัญหาที่เกิดขึ้นบ่อยๆ ในการเขียนโปรแกรม. การเรียนรู้ Patterns จะช่วยให้เราสามารถออกแบบ **Components** ในระดับ C3 ให้มีโครงสร้างที่ดี, ยืดหยุ่น, และดูแลรักษาง่ายในระยะยาว.

เป้าหมายของ C3 คือการใช้ Patterns เหล่านี้เพื่อสร้างแผนผังส่วนประกอบ (เช่น Authentication Service, WebSocket Manager) ภายในแต่ละ Container ซึ่งจะเป็นพิมพ์เขียวสุดท้ายที่ทีมพัฒนาจะนำไปใช้ในการแบ่งงานและลงมือเขียนโค้ด.
