Backend Journey: Why, What & Next?

สรุปแนวคิดสำคัญของการเรียนรู้วันที่ 4: ทำไมเราต้องใช้เทคโนโลยีเหล่านี้, มันทำอะไรได้บ้าง, และจะนำไปประยุกต์ใช้อย่างไร

จุดเริ่มต้น: ข้อจำกัดของ Frontend

JavaScript ทำงานได้แค่ในเบราว์เซอร์ จัดการข้อมูลถาวรไม่ได้

ทางออก: สร้าง Backend Server

สร้าง "สมอง" ของระบบที่ทำงานเบื้องหลัง เพื่อจัดการข้อมูลและ Logic ที่ซับซ้อน

เป้าหมาย: Full-stack Application

เชื่อมต่อ Frontend และ Backend เข้าด้วยกันเพื่อสร้างเว็บแอปพลิเคชันที่สมบูรณ์

1. Node.js & NPM

การวางรากฐานของ Backend

Node.js Runtime

ทำไมต้องมี?

JavaScript ถูกจำกัดให้ทำงานได้แค่ในเบราว์เซอร์ ไม่สามารถเข้าถึงไฟล์หรือสร้างเซิร์ฟเวอร์ได้

มีไว้ทำอะไร?

เป็น "สภาพแวดล้อม" ที่ปลดปล่อยให้ JavaScript ทำงานบนเซิร์ฟเวอร์ได้ ทำให้มันกลายเป็นภาษาสำหรับ Backend

ประยุกต์ใช้ต่อไป?

เป็นรากฐานในการติดตั้งและรัน Express.js ซึ่งเป็น Web Framework ที่เราจะใช้สร้าง API

NPM

ทำไมต้องมี?

การเขียนฟังก์ชันทุกอย่างด้วยตัวเองตั้งแต่ต้น (เช่น การสร้างเซิร์ฟเวอร์, การจัดการรหัสผ่าน) เป็นเรื่องที่ช้าและซับซ้อน

มีไว้ทำอะไร?

เป็น "คลังเครื่องมือ" ที่ให้นักพัฒนาแบ่งปันและดาวน์โหลดโค้ด (Package) สำเร็จรูปมาใช้งานได้ทันที

ประยุกต์ใช้ต่อไป?

เราใช้ NPM เพื่อติดตั้ง Express.js, CORS, Nodemon และเครื่องมืออื่นๆ ที่จำเป็นทั้งหมดสำหรับโปรเจคของเรา

2. Express.js

การสร้างโครงสร้างเซิร์ฟเวอร์ที่แข็งแกร่ง

Express Framework

ทำไมต้องมี?

การสร้างเซิร์ฟเวอร์ด้วย `http` module ของ Node.js โดยตรงนั้นต้องเขียนโค้ดเยอะและจัดการได้ยากเมื่อโปรเจคใหญ่ขึ้น

มีไว้ทำอะไร?

เป็น "โครงสร้างสำเร็จรูป" สำหรับสร้างเว็บเซิร์ฟเวอร์ ช่วยลดความซับซ้อนและจัดระเบียบโค้ดให้เป็นมาตรฐาน

ประยุกต์ใช้ต่อไป?

เป็นแกนหลักของ Backend ทั้งหมด เราจะใช้มันเพื่อสร้าง API endpoints ทั้งหมดในโปรเจค

Routing

ทำไมต้องมี?

การใช้ `if/else` เพื่อจัดการ URL ที่แตกต่างกันนั้นไม่เป็นระเบียบและดูแลรักษายากเมื่อมี URL จำนวนมาก

มีไว้ทำอะไร?

เป็น "สารบัญ" ของ API ช่วยให้เรากำหนดเส้นทาง (เช่น `/users`, `/products/:id`) ได้อย่างชัดเจนและเป็นระเบียบ

ประยุกต์ใช้ต่อไป?

เราจะสร้าง Routes สำหรับจัดการข้อมูลเมนูอาหาร (CRUD) เพื่อให้ React App สามารถเรียกใช้งานได้

Middleware

ทำไมต้องมี?

มีงานหลายอย่างที่ต้องทำซ้ำๆ กับทุก Request (เช่น การตรวจสอบสิทธิ์, การแปลงข้อมูล) การเขียนโค้ดเหล่านี้ในทุก Route ทำให้โค้ดซ้ำซ้อน

มีไว้ทำอะไร?

เป็น "ด่านตรวจ" ที่ทำงานก่อนถึง Logic หลัก ช่วยให้เราจัดการงานที่ทำซ้ำๆ ในที่เดียว เช่น `express.json()` เพื่ออ่านข้อมูล, `cors()` เพื่อจัดการความปลอดภัย

ประยุกต์ใช้ต่อไป?

เราจะใช้ `cors()` เพื่ออนุญาตให้ React App ที่อยู่คนละ Port สามารถเชื่อมต่อกับ API ของเราได้

3. JSON & REST API

การสร้างสะพานเชื่อมเพื่อสื่อสารและจัดการข้อมูล

JSON

ทำไมต้องมี?

Frontend (ที่รันในเบราว์เซอร์) และ Backend (ที่รันบนเซิร์ฟเวอร์) ต้องมี "ภาษากลาง" ที่เข้าใจตรงกันเพื่อแลกเปลี่ยนข้อมูล

มีไว้ทำอะไร?

เป็นรูปแบบข้อมูลแบบ Text ที่เป็นมาตรฐานสากล ทำให้ทุกระบบสามารถส่งและรับข้อมูลในรูปแบบเดียวกันได้

ประยุกต์ใช้ต่อไป?

ข้อมูลเมนูอาหารทั้งหมดที่ส่งระหว่าง React App และ Express API ของเราจะอยู่ในรูปแบบของ JSON

REST API

ทำไมต้องมี?

หากไม่มีมาตรฐานในการออกแบบ API แต่ละคนจะสร้าง Endpoint ในรูปแบบของตัวเอง (เช่น `/getUser`, `/fetchUsers`) ทำให้สับสนและใช้งานยาก

มีไว้ทำอะไร?

เป็น "แบบแผน" ในการออกแบบ API ที่เป็นที่ยอมรับทั่วโลก โดยใช้ URL สื่อถึง "ทรัพยากร" และใช้ HTTP Method สื่อถึง "การกระทำ"

ประยุกต์ใช้ต่อไป?

API ของเราจะถูกออกแบบตามหลัก REST ทำให้ React App สามารถเรียกใช้ Endpoint (เช่น `GET /api/foods`) ได้อย่างเป็นมาตรฐาน

CRUD Operations

ทำไมต้องมี?

เราต้องการชุดคำสั่งที่เป็นมาตรฐานสำหรับการจัดการข้อมูล ไม่ว่าจะเป็นการสร้าง, อ่าน, แก้ไข หรือลบ

มีไว้ทำอะไร?

คือชุดการดำเนินการ 4 อย่าง (Create, Read, Update, Delete) ที่ครอบคลUMการจัดการข้อมูลทั้งหมด ซึ่งถูกจับคู่กับ HTTP Method (POST, GET, PUT, DELETE)

ประยุกต์ใช้ต่อไป?

React App ของเราจะมีฟีเจอร์สำหรับ เพิ่ม, ดู, แก้ไข, และลบเมนูอาหาร โดยการเรียก API ที่มี CRUD Operations เหล่านี้
