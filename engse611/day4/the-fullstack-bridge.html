โลกสองใบ: Client vs. Server

การพัฒนาเว็บสมัยใหม่แบ่งออกเป็น 2 ส่วนหลักๆ ที่ทำงานร่วมกันแต่มีหน้าที่แตกต่างกันอย่างสิ้นเชิง

Client-Side (ฝั่งผู้ใช้)

  • ที่อยู่: ทำงานบนเว็บเบราว์เซอร์ของผู้ใช้
  • หน้าที่: แสดงผลหน้าตาเว็บ (UI) และตอบสนองต่อผู้ใช้
  • หน่วยความจำ: ชั่วคราว (State) ข้อมูลหายเมื่อปิดเว็บ
  • ข้อจำกัด: เข้าถึงฐานข้อมูลโดยตรงไม่ได้

Server-Side (ฝั่งเรา)

  • ที่อยู่: ทำงานบนคอมพิวเตอร์เซิร์ฟเวอร์
  • หน้าที่: จัดการ Logic และเป็นแหล่งข้อมูลจริง (Source of Truth)
  • หน่วยความจำ: ถาวร เชื่อมต่อฐานข้อมูล/ไฟล์ได้
  • พลังพิเศษ: เข้าถึงไฟล์, ฐานข้อมูล, ระบบอื่นๆ ได้
คำถามสำคัญ: แล้วโลกสองใบนี้... คุยกันได้อย่างไร?

สะพานเชื่อม: กลไกการทำงานของ API

เพื่อให้ทั้งสองฝั่งสื่อสารกันได้ เราต้องสร้าง "สะพาน" ที่เรียกว่า **API (Application Programming Interface)** โดยใช้ HTTP เป็นช่องทางในการส่ง "คำขอ" (Request) และ "คำตอบ" (Response)

1. React App ส่งคำขอ (Request)

เมื่อผู้ใช้คลิก, React จะใช้ fetch() ส่งคำขอ HTTP ไปยัง URL ของ Backend

2. Express Server รับคำขอ

เซิร์ฟเวอร์ Node.js ของเราดักฟัง Request และ Express Router จะจับคู่ URL ที่เข้ามา

3. Server ประมวลผล

Controller Function ทำงานตาม Logic เช่น อ่านข้อมูลจากไฟล์ foods.json

4. Server ส่งคำตอบ (Response)

เซิร์ฟเวอร์ส่งข้อมูลกลับไปในรูปแบบ JSON พร้อมกับ Status Code (เช่น 200 OK)

5. React App อัปเดตหน้าเว็บ

React รับข้อมูล JSON มาอัปเดต State ทำให้หน้าเว็บ (UI) เปลี่ยนแปลงอัตโนมัติ

ข้อตกลงในการสื่อสาร: Frontend vs. Backend

เพื่อให้การเชื่อมต่อราบรื่น ทั้งสองฝั่งต้องมี "ข้อตกลง" หรือหน้าที่ความรับผิดชอบของตัวเอง

หน้าที่ของ Frontend

  • ส่งคำขอ (Request): เรียก API Endpoint ที่ถูกต้อง พร้อมส่งข้อมูลตามรูปแบบที่กำหนด
  • รับคำตอบ (Response): รอรับข้อมูล JSON และจัดการข้อมูลที่ได้มา
  • จัดการสถานะ UI: แสดงสถานะ "กำลังโหลด" ระหว่างรอ และอัปเดตหน้าเว็บเมื่อได้รับข้อมูล
  • จัดการข้อผิดพลาด: เตรียมพร้อมรับมือเมื่อ Backend ส่ง Error กลับมา และแสดงข้อความให้ผู้ใช้ทราบ

หน้าที่ของ Backend

  • รับคำขอ (Request): สร้างและกำหนด Endpoint เพื่อรอรับคำขอจาก Frontend
  • ตรวจสอบข้อมูล (Validate): "ไม่เชื่อใจข้อมูลจาก Client" ต้องตรวจสอบความถูกต้องของข้อมูลที่ส่งมาเสมอ
  • ประมวลผล Logic: จัดการข้อมูล (CRUD) ตามคำขอ เช่น บันทึกลงฐานข้อมูล
  • ส่งคำตอบ (Response): ส่งข้อมูลกลับในรูปแบบ JSON พร้อม Status Code ที่ถูกต้องและสม่ำเสมอ
  • จัดการ CORS: กำหนดนโยบายเพื่ออนุญาตให้ Frontend ที่อยู่คนละ Origin สามารถเรียกใช้ API ได้

กรณีศึกษา: เว็บรีวิวร้านอาหาร

เพื่อให้เห็นภาพชัดเจน ลองดูตัวอย่าง Flow การทำงานเมื่อผู้ใช้ต้องการดูข้อมูลร้านอาหารในโปรเจคที่เราจะทำกัน

1. User Action

ผู้ใช้คลิกที่ "ร้านอาหาร A" ใน React App, UI แสดงสถานะ "กำลังโหลด..."

2. Request

React ส่ง GET request ไปยัง /api/restaurants/1

3. Backend Process

Express Server รับ request, ไปดึงข้อมูลร้านอาหาร id=1 จากไฟล์หรือฐานข้อมูล

4. Response

Server ส่งข้อมูลร้านอาหารกลับไปเป็น JSON: { "name": "ร้านอาหาร A", ... }

5. UI Update

React รับข้อมูล, อัปเดต State, และแสดงผลรายละเอียดของร้านอาหารบนหน้าจอ
