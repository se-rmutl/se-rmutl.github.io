Core framework
Asset → Threat → Vulnerability → Risk
นี่คือโครงคิดพื้นฐานที่สุดของการทำ Security Analysis และใช้ได้ตั้งแต่ระบบนักศึกษาไปจนถึง production enterprise
Slide 6 • Core model
Assetสิ่งที่มีคุณค่า เช่น accounts, source code, database, logs
→
Threatสิ่งที่อาจก่อความเสียหาย เช่น attacker, malware, insider
→
Vulnerabilityจุดอ่อนที่ถูกโจมตีได้ เช่น weak password, buggy code
→
Riskโอกาส × ผลกระทบ ที่เกิดขึ้นจริงกับระบบ

ตัวอย่าง Mobile Banking

  • Asset: เงิน, transaction, OTP, PII
  • Threat: phishing, malware, credential stuffing
  • Vulnerability: reused password, weak MFA, API bug
  • Risk: account takeover และเงินสูญหาย
ถามตัวเอง: ถ้าคุณสร้างระบบลงทะเบียนเรียนของมหาวิทยาลัย สินทรัพย์ที่สำคัญที่สุดคืออะไร? เกรด? ข้อมูลนักศึกษา? สิทธิ์ของอาจารย์?
หาก framework นี้ชัด การวิเคราะห์อย่างอื่นจะตามมาได้ง่ายThink in systems
CIA Triad
แกนกลางของ Information Security: C • I • A
Confidentiality, Integrity, Availability ช่วยให้เรามองผลกระทบของการโจมตีได้ครบทุกมิติ
Slide 7 • CIA
Confidentiality
ข้อมูลต้องไม่รั่วไปถึงคนที่ไม่มีสิทธิ์
Integrity
ข้อมูลต้องถูกต้อง ไม่ถูกแก้ไขโดยมิชอบ
Availability
ระบบและข้อมูลต้องพร้อมใช้งานเมื่อจำเป็น

ตัวอย่าง C

ข้อมูลผู้ป่วยหลุด, source code รั่ว, password plaintext ถูกเห็นได้

ตัวอย่าง I

เกรดถูกแก้, transaction ถูกเปลี่ยน, log ถูกปลอม

ตัวอย่าง A

ระบบล่ม, ransomware เข้ารหัสไฟล์, DDoS ทำให้เว็บเข้าไม่ได้

ทุกช่องโหว่และทุก control ควรถูกถามว่ากระทบ C, I หรือ A อย่างไรAnchor principle

Week 3
Vulnerability: จุดอ่อนของระบบที่ Threat ใช้ประโยชน์ได้
ความต่างระหว่าง threat กับ vulnerability ต้องชัด เพราะหนึ่งคือ “ผู้ก่อเหตุ/สิ่งคุกคาม” อีกหนึ่งคือ “ช่องให้เข้าโจมตี”
Slide 13 • Vulnerabilities

Software

  • Buffer overflow
  • SQL Injection
  • XSS
  • Authentication bypass
  • Insecure deserialization

Configuration

  • Default credentials
  • Unnecessary services
  • Weak encryption
  • Missing patches

Human-related

  • Weak passwords
  • Lack of awareness
  • Insider threats
  • ตกเป็นเหยื่อ social engineering
Remember: Threat = “โจร”, Vulnerability = “ประตูล็อกไม่ดี”, Risk = โอกาสที่โจรจะใช้ประตูนี้เข้ามาก่อความเสียหาย
หลาย incident ใหญ่เริ่มจากช่องโหว่พื้นฐานที่ถูกมองข้ามBasics matter
CVE / NVD / CVSS
จากการค้นพบช่องโหว่ → การสื่อสารมาตรฐาน → การจัดลำดับความรุนแรง
นักพัฒนาสมัยใหม่ควรอ่าน CVE ได้, ดู NVD เป็น, และตีความ CVSS ให้สัมพันธ์กับบริบทของระบบตนเอง
Slide 14 • Vulnerability pipeline

→
NVDฐานข้อมูลรายละเอียด, impact, references
→
CVSSคะแนนความรุนแรง 0.0–10.0

Actionpatch, mitigate, prioritize

ตัวอย่างจากบทเรียน

  • Heartbleed: memory over-read ใน OpenSSL
  • EternalBlue: SMB vulnerability ที่เชื่อมกับ WannaCry
  • Log4Shell: RCE ระดับ critical ใน supply chain สำคัญ
  • XZ Utils Backdoor: เตือนเรื่อง supply-chain security
ข้อสังเกต: CVSS สูงไม่ได้แปลว่าต้องแก้ก่อนเสมอ หาก service นั้นไม่ได้ expose หรือไม่มี business impact สูงใน environment ของเรา
อย่าอ่าน CVSS แบบตัดขาดจากบริบทการใช้งานจริงSeverity ≠ priority เสมอไป

Crypto choices
Symmetric vs Asymmetric Encryption
ทั้งสองแบบมีหน้าที่ต่างกัน และระบบจริงมักใช้ร่วมกันแบบ hybrid
Slide 26 • Encryption models

Symmetric

  • ใช้ key เดียวกันในการเข้ารหัสและถอดรหัส
  • เร็วมาก เหมาะกับข้อมูลขนาดใหญ่
  • ปัญหาหลัก: key distribution
  • ตัวอย่าง: AES, ChaCha20

Asymmetric

  • ใช้ public / private key
  • แก้ปัญหาแลก key และใช้ทำ digital signature ได้
  • ช้ากว่าและกินทรัพยากรมากกว่า
  • ตัวอย่าง: RSA, ECC, Diffie-Hellman
Hybrid Encryption: ใน TLS/HTTPS มักใช้ asymmetric เพื่อแลก session key แล้วใช้ symmetric เข้ารหัสข้อมูลจริง เพราะเร็วกว่า
เลือก crypto model ให้ตรงงาน แทนที่จะใช้เพราะชื่อคุ้นFit for purpose
Integrity & trust
Hash, MAC, Digital Signature และ PKI
นี่คือส่วนที่ทำให้เราตรวจสอบความถูกต้องของข้อมูลและตัวตนของผู้ส่งได้
Slide 27 • Integrity tools

Hash

สรุปข้อมูลเป็น digest; เปลี่ยนข้อมูลนิดเดียว hash เปลี่ยนมาก

MAC

เพิ่ม shared secret เพื่อยืนยันทั้ง integrity และ authenticity

Digital Signature

ใช้ private key ลงนาม และ public key ตรวจสอบ

PKI / Certificate

สร้าง chain of trust ว่า public key นี้เป็นของใคร

มุมคิดแบบวิศวกร

Hash ≠ encryption, Base64 ≠ security, และการเก็บรหัสผ่านควรใช้ password hashing ที่เหมาะสม ไม่ใช่ reversible encryption

ถาม: ถ้า hash ถูกต้อง แต่ไฟล์มาจากคนที่เราไม่รู้จัก เราไว้ใจที่มาได้จริงหรือยัง?

เน้นป้องกันการเข้าถึง แก้ไข หรือทำลายข้อมูล/ระบบโดยไม่ได้รับอนุญาต

ConfidentialityIntegrityAvailability

Privacy

เน้นสิทธิ์ของเจ้าของข้อมูล ว่าจะเก็บ ใช้ เปิดเผย ส่งต่อ และลบอย่างไรอย่างเป็นธรรมและโปร่งใส

PurposeConsentControl
ข้อสังเกต: ระบบอาจ “secure” มาก แต่ยัง “ละเมิด privacy” ได้ หากเก็บข้อมูลเกินจำเป็น ติดตามผู้ใช้เกินวัตถุประสงค์ หรือทำให้การถอน consent ยากเกินไป
Good software ต้องดีทั้ง security และ privacyBoth matter
PDPA
PDPA สำหรับ Developer: จากตัวบทสู่การตัดสินใจในระบบ
เมื่อระบบเก็บ personal data นักพัฒนาไม่สามารถโยนทั้งหมดให้ฝ่ายกฎหมายได้ เพราะ code คือที่ที่นโยบายถูกทำให้เป็นจริง
Slide 35 • PDPA

Lawful Basis ที่ควรรู้

  • Contract — จำเป็นต่อการให้บริการ
  • Consent — ต้องชัดเจน แยกวัตถุประสงค์
  • Legitimate Interest — ต้องชั่งดุลผลประโยชน์และสิทธิ์

Data Subject Rights

  • ขอเข้าถึงข้อมูล
  • แก้ไขข้อมูล
  • ลบข้อมูล / เพิกถอน consent
  • คัดค้าน/จำกัดการประมวลผลบางกรณี
สิ่งที่ developer ควรทำใน code: บันทึก consent อย่างตรวจสอบได้, มี retention policy, ทำ audit log, รองรับ data subject request, และเข้ารหัสข้อมูลสำคัญทั้ง at rest / in transit
ถาม: ถ้า form สมัครสมาชิกขอวันเกิด, location, contact list, และ device ID ทั้งที่ไม่จำเป็นต่อการใช้งาน นี่คือ design ที่ดีหรือไม่?
Compliance ที่ดีเกิดจาก product decisions และ system design ที่ดีCode is policy
Privacy by Design
ออกแบบ Privacy ตั้งแต่ต้น ไม่ใช่ติดเพิ่มทีหลัง
แนวคิดนี้ทำให้ระบบ “เคารพผู้ใช้โดยค่าเริ่มต้น” และลดต้นทุนการแก้ย้อนหลัง
Slide 36 • PbD

Data minimization

เก็บเฉพาะสิ่งที่จำเป็นจริงต่อวัตถุประสงค์

Privacy as default

ค่าตั้งต้นควรปกป้องผู้ใช้สูงสุด ไม่ใช่เปิดทุกอย่างแล้วให้ปิดเอง

User control

ถอน consent, export data, delete account ได้จริง

Transparency

อธิบายด้วยภาษาคน ไม่ซ่อนใน dark patterns

Security built-in

Encrypt, access control, auditability เป็นส่วนหนึ่งของ design

Lifecycle thinking

คิดตั้งแต่เก็บ ใช้ แชร์ เก็บรักษา จนลบข้อมูล

Dark patterns ที่ควรหลีกเลี่ยง: ปุ่ม “ยอมรับทั้งหมด” ใหญ่เด่น แต่ “ปฏิเสธ” ซ่อนลึก, บังคับ all-or-nothing consent, หรือทำให้ลบบัญชียากกว่าสมัคร
Modern UX insight: ประสบการณ์ที่ดีไม่ใช่การบังคับผู้ใช้ แต่คือการทำให้ผู้ใช้เข้าใจและควบคุมข้อมูลตัวเองได้
Privacy-friendly design คือ competitive advantage ไม่ใช่ภาระอย่างเดียวRespect users
Ethics
Ethics for Software Engineers: ทำได้ ≠ ควรทำ
เมื่อซอฟต์แวร์มีอำนาจต่อผู้ใช้มากขึ้น วิศวกรก็ต้องมีวิจารณญาณมากขึ้นเช่นกัน
Slide 37 • Ethics

Ethics Decision Lens

  • ใครได้ประโยชน์ และใครอาจเสียหาย?
  • ผู้ใช้เข้าใจจริงหรือไม่ว่าระบบกำลังเก็บ/อนุมานอะไร?
  • หากข้อมูลรั่วหรือโมเดลมี bias ใครจะรับผลกระทบ?
  • มีทางเลือกที่ปลอดภัยและเป็นธรรมกว่านี้หรือไม่?

ประเด็นตัวอย่างจาก lecture

  • Location tracking ที่เกินจำเป็น
  • Behavioral profiling โดยไม่โปร่งใส
  • AI bias ในการคัดคน / scoring
ข้อคิดจาก ACM/IEEE spirit: วิศวกรซอฟต์แวร์มีหน้าที่ต่อสาธารณะ ไม่ใช่แค่ต่อ requirement document
Ethics ไม่ใช่วิชาแยก แต่คือส่วนหนึ่งของการตัดสินใจทางวิศวกรรมProfessional responsibility
Apply to SDLC
จากบทเรียนทั้งวิชา → สู่ Secure SDLC และ DevSecOps
นี่คือสไลด์ที่แปลงความรู้ทั้งหมดให้เป็น workflow ใช้งานจริงในการทำโปรเจกต์

Slide 39 • Final
1. Asset สำคัญที่สุดของระบบคืออะไร?
2. ใครคือ attacker หรือ misuse actor ที่เป็นไปได้?
3. ช่องโหว่ของ design / code / config อยู่ตรงไหน?
4. ผลกระทบต่อ C, I, A จะเป็นอย่างไร?
5. ใครควรเข้าถึงข้อมูลอะไรได้บ้าง?
6. Input / output / session / secrets ปลอดภัยพอหรือยัง?
7. มี logging และ audit trail พอสำหรับ incident หรือไม่?
8. เก็บ personal data เกินจำเป็นหรือเปล่า?
9. ผู้ใช้สามารถควบคุม consent และข้อมูลตนเองได้จริงไหม?
10. ถ้าระบบนี้มีปัญหา เราอธิบายต่อผู้ใช้และอาจารย์ได้ไหมว่าปกป้องเขาอย่างไร?
Final thought: นักพัฒนาที่ดีไม่ใช่คนที่ทำระบบ “เสร็จ” แต่คือคนที่ทำระบบให้ใช้งานได้ ปลอดภัย เชื่อถือได้ และเคารพผู้ใช้
