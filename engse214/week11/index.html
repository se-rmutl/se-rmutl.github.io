🔄 ความสัมพันธ์: Security vs Privacy

Security ดี ≠ Privacy ดี | แต่ Security แย่ → Privacy จะแย่เสมอ

Sec: ✅ Priv: ❌

เข้ารหัสอย่างดี แต่เอาไปขาย

ระบบ Encrypted DB แน่นหนา แฮกเกอร์เจาะไม่ได้ แต่บริษัทแอบเอาพฤติกรรมผู้ใช้ไปขายให้ Data Broker โดยไม่บอก

Sec: ❌ Priv: ✅

นโยบายเยี่ยม แต่ระบบกาก

Privacy Policy โปร่งใส ขอ Consent ชัดเจน แต่ Database กาารจัดเก็บไม่มี Password ใครก็โหลดไปดูได้

Sec: ✅ Priv: ✅

สมบูรณ์แบบ (Ideal)

เข้ารหัสข้อมูลอย่างดี + มี Access Control ภายในองค์กร + ขอ Consent ชัดเจนและใช้ตามวัตถุประสงค์เท่านั้น

มิติ (CIA) มุมมอง Security ส่วนขยาย Privacy Framework
Confidentiality เฉพาะผู้มีสิทธิ์ (Admin/System) เท่านั้นเข้าถึงได้ + เจ้าของข้อมูลเป็นคน "กำหนด" ว่าใครควรมีสิทธิ์
Integrity ข้อมูลไม่ถูกแก้ไขโดยแฮกเกอร์ + ข้อมูลต้องถูกต้องตรงตามจริง และเจ้าของขอแก้ไขได้
Availability ระบบไม่ล่ม พร้อมให้งานองค์กรเสมอ + เจ้าของข้อมูลต้องสามารถเข้าถึงหรือขอดูข้อมูลตัวเองได้เสมอ

Role 1: Data Subject (เจ้าของข้อมูล)

สิทธิ์ 6 ประการ ที่ UI/UX หรือ System ต้องรองรับ:

Right of Access

สิทธิขอเข้าถึงและขอรับสำเนาข้อมูลของตน (เช่น ปุ่ม Download My Data)

Right to Rectification

สิทธิขอแก้ไขข้อมูลให้ถูกต้อง เป็นปัจจุบัน (ฟอร์ม Edit Profile)

Right to Erasure

สิทธิขอให้ลบข้อมูล หรือ Right to be Forgotten (ปุ่ม Delete Account)

Right to Object

สิทธิคัดค้านการประมวลผล (เช่น Unsubscribe Marketing Emails)

Right to Data Portability

สิทธิขอโอนย้ายข้อมูลไปแพลตฟอร์มอื่นในรูปแบบไฟล์ที่อ่านได้ (CSV, JSON)

Right to Complain

ร้องเรียนต่อคณะกรรมการ (PDPC) หากบริษัทไม่ทำตาม

Role 2: Data Controller (ผู้ควบคุมข้อมูล)

หน้าที่หลักที่ Developer ต้องสร้างระบบมารองรับ:

  • สร้างกลไก ขอ Consent อย่างชัดเจน พร้อมบันทึก Consent Log ว่ากดยอมรับเมื่อไหร่/เวอร์ชันไหน
  • แจ้งวัตถุประสงค์ (Privacy Notice) ให้ผู้ใช้ทราบ ก่อนหรือขณะ เก็บข้อมูล
  • วางมาตรการ Security ตามมาตรฐาน (Encryption, Access Control)
  • Data Breach Notification: ระบบต้องแจ้งเตือนได้ภายใน 72 ชั่วโมง หากข้อมูลหลุด
  • เขียน CRON Job หรือ Script ตาม Data Retention Policy (ลบทิ้งเมื่อหมดอายุ/ไม่ใช้แล้ว)

Role 3: Data Processor (ผู้ประมวลผลข้อมูล)

Cloud, Payment Gateway, 3rd Party API

ข้อจำกัดทางกฎหมาย:

  • ประมวลผลข้อมูล ตามคำสั่ง (Instruction) ของ Controller เท่านั้น
  • ห้ามแอบนำข้อมูลไปใช้งานเองนอกเหนือจากสัญญา (DPA)
  • ต้องมีมาตรการรักษาความปลอดภัย (Security Measures) ขั้นสูง
  • แจ้ง Controller ทันทีหาก Cloud ของตนเองถูก Hack

ฝังลึกอยู่ในโครงสร้าง Architecture เป็นเนื้อเดียวกับซอฟต์แวร์

4 Positive-Sum

ไม่ต้องเลือกระหว่าง Security vs Privacy ต้องได้ทั้งคู่แบบ Win-Win (Full Functionality)

5 End-to-End Security

คุ้มครองตลอดวงจรชีวิต (ตั้งแต่เก็บข้อมูล ประมวลผล ยันทำลายทิ้ง)

6 Visibility & Transparency

โปร่งใส ตรวจสอบได้ว่าใครเอาข้อมูลไปทำอะไรบ้าง (Auditability)

7 Respect for User Privacy

เคารพสิทธิผู้ใช้เป็นศูนย์กลาง (User-Centric) ออกแบบ UX ให้คนเข้าใจง่าย

การนำไปโค้ดดิ้ง (SE Implementation)

1. Data Minimization (เก็บเท่าที่จำเป็น)

ตอนออกแบบ API SELECT * FROM users ไม่ควรใช้พร่ำเพรื่อ ดึงมาเฉพาะฟิลด์ที่ UI ต้องการจริงๆ ป้องกันข้อมูลหลุดเกินจำเป็น

2. Data Masking & Anonymization

ตอนแสดงผลหน้าจอ: แสดงเบอร์โทรแบบ 081-XXX-XX99
ตอนเก็บลง DB: ใช้การทำ Hashing ข้อมูลที่ระบุตัวตน (PII)

3. Automated Data Deletion

เขียน Worker/Cron job เพื่อรัน DELETE FROM logs WHERE created_at < NOW() - 90 days อัตโนมัติ ตาม Retention Policy

Ethics in Software Engineering

จริยธรรมของวิศวกรซอฟต์แวร์ เมื่อกฎหมายตามเทคโนโลยีไม่ทัน

ACM Code of Ethics (จรรยาบรรณวิชาชีพ)

องค์กรระดับโลก ACM กำหนดจรรยาบรรณที่เกี่ยวข้องกับ Privacy ไว้ชัดเจน:

1.2 Avoid harm (หลีกเลี่ยงการสร้างความเสียหาย)

การละเมิดความเป็นส่วนตัว ไม่ใช่แค่เรื่องในคอมพิวเตอร์ แต่มันนำไปสู่ความเสียหายทั้ง ทรัพย์สิน, สภาพจิตใจ, และชื่อเสียง ของผู้ใช้งานในโลกจริง

1.6 Respect privacy (เคารพความเป็นส่วนตัว)

วิศวกรต้องเก็บข้อมูลให้ น้อยที่สุด ปกป้องข้อมูลให้ ดีที่สุด และทำระบบให้ โปร่งใสที่สุด ต่อผู้ใช้งาน

สรุปท้ายบท (Conclusion)

  • Security ปกป้องระบบ แต่ Privacy ปกป้องสิทธิ์คน (ต้องมีทั้งคู่)
  • PDPA / GDPR บังคับให้ระบบต้องรองรับสิทธิของ Data Subject
  • Privacy by Design ฝัง Privacy ไว้ใน Architecture ตั้งแต่วันแรก
  • จริยธรรม (Ethics) คือเกราะป้องกันชั้นสุดท้าย เมื่อกฎหมายตามไม่ทัน
