Part 1: Security vs Privacy

ความแตกต่างที่วิศวกรซอฟต์แวร์ต้องเข้าใจ

⚖️ ความหมายและความแตกต่าง

"ผู้เรียนหลายคนมักสับสนว่า Security กับ Privacy คือสิ่งเดียวกัน"

แต่ในความเป็นจริงทั้งสองมีมุมมองที่ แตกต่างกันโดยพื้นฐาน
การทำระบบให้ "ปลอดภัย" ไม่ได้แปลว่าระบบนั้นเคารพ "ความเป็นส่วนตัว" เสมอไป

🛡️ SECURITY (ความมั่นคงปลอดภัย)

  • เป้าหมาย: ปกป้องระบบ & ข้อมูล จากผู้ไม่ได้รับอนุญาต
  • มุมมอง: ผู้ดูแลระบบ (System Admin / Developer)
  • คำถามหลัก: "ใครสามารถเข้าถึงข้อมูลได้?"

🛠️ เครื่องมือที่ใช้

Firewall Authentication Encryption IDS/IPS

👁️ PRIVACY (ความเป็นส่วนตัว)

  • เป้าหมาย: ควบคุมข้อมูลส่วนตัวของคน ว่าใครเข้าถึงได้บ้าง
  • มุมมอง: เจ้าของข้อมูล (Data Subject)
  • คำถามหลัก: "เจ้าของข้อมูลต้องการให้ใครเห็น?"

🛠️ เครื่องมือที่ใช้

Consent Privacy Policy Anonymization PDPA/GDPR

🔄 OVERLAP ที่สำคัญมาก

Security ดี ≠ Privacy ดี
แต่ Security ไม่ดี → Privacy ก็ไม่ดีด้วย

ทั้งคู่จำเป็นต้องทำงานร่วมกันอย่างขาดไม่ได้

❌ ขาด Privacy

Encrypted DB อย่างดี แต่บริษัทแอบเอาข้อมูลไปขายให้ Data Broker

Security ✓ | Privacy ✗

❌ ขาด Security

Privacy Policy ดีงาม ขอ Consent ถูกต้อง แต่ DB โดน Hack เพราะไม่มี Password

Privacy ✓ | Security ✗

✅ สมบูรณ์แบบ

เข้ารหัสข้อมูล + ขอ Consent ชัดเจน + จำกัด Access Control ภายในองค์กร

Security ✓ | Privacy ✓

🔺 ความสัมพันธ์กับ CIA Triad

มิติ CIA Triad (Security) Privacy Framework (ส่วนขยาย)
Confidentiality เฉพาะผู้มีสิทธิ์เข้าถึงได้ + เจ้าของข้อมูลควบคุมว่าใครมีสิทธิ์
Integrity ข้อมูลไม่ถูกแก้ไขโดยไม่ได้รับอนุญาต + ข้อมูลต้องถูกต้อง เจ้าของขอแก้ไขได้
Availability ระบบพร้อมใช้งานเสมอ + เจ้าของข้อมูลเข้าถึงข้อมูลตัวเองได้

💡 บทสรุป: ส่วนขยายของ CIA

Privacy คือ "ส่วนขยาย" ของ Confidentiality ในมุมมองของเจ้าของข้อมูล

ไม่ใช่แค่ว่า "คนอื่นเข้าถึงไม่ได้"
แต่รวมถึง "เจ้าของข้อมูลมีอำนาจควบคุมด้วย"

⚠️ ความเข้าใจผิดที่พบบ่อยของ Developer (1/3)

"เข้ารหัสข้อมูลไว้แล้ว ก็ปลอดภัยและส่วนตัวแล้ว"

ปัญหาที่แท้จริง:

ถ้า Application หรือบริษัทของเราสามารถถอดรหัส (Decrypt) และนำข้อมูลไปวิเคราะห์เพื่อขายให้ Advertiser ได้ นั่นคือระบบมี Security ที่ดี แต่มี Privacy ที่แย่มากสำหรับผู้ใช้

⚠️ ความเข้าใจผิดที่พบบ่อยของ Developer (2/3)

"มี Privacy Policy ยาว 10 หน้าก็พอแล้ว"

ปัญหาที่แท้จริง:

กฎหมาย PDPA/GDPR ต้องการสิ่งที่เรียกว่า "Informed Consent" ผู้ใช้ต้องเข้าใจจริงๆ ด้วยภาษาที่อ่านง่าย ชัดเจน ก่อนที่จะกดยินยอม การซ่อนเงื่อนไขในข้อความยาวๆ ถือว่าผิดหลักการ

⚠️ ความเข้าใจผิดที่พบบ่อยของ Developer (3/3)

"ข้อมูลใน Database บริษัทเรา ใช้ยังไงก็ได้"

ปัญหาที่แท้จริง:

ข้อมูลส่วนบุคคลเป็นกรรมสิทธิ์ของ เจ้าของข้อมูล (Data Subject) ไม่ใช่ของบริษัท บริษัทมีสถานะเพียง "ผู้ดูแล" ที่ต้องใช้งานตามวัตถุประสงค์ที่ได้แจ้งและขออนุญาตไว้เท่านั้น

👤 Data Subject (เจ้าของข้อมูล)

สิทธิ์ที่มีตามกฎหมาย:

  • สิทธิในการเข้าถึงข้อมูล (Right of Access)
  • สิทธิในการแก้ไขข้อมูลให้ถูกต้อง (Right to Rectification)
  • สิทธิในการลบข้อมูล (Right to Erasure / Right to be Forgotten)
  • สิทธิในการคัดค้านการประมวลผล (Right to Object)
  • สิทธิในการขอโอนย้ายข้อมูล (Right to Data Portability)
  • สิทธิร้องเรียนต่อคณะกรรมการ PDPC

🏢 Data Controller (ผู้ควบคุมข้อมูล)

ในฐานะ Developer
เรามักสร้างระบบให้กลุ่มนี้

หน้าที่หลักที่ระบบต้องรองรับ:

  • แจ้งวัตถุประสงค์การใช้งานอย่างชัดเจน
  • ขอ Consent (ความยินยอม) และบันทึกประวัติการขอ
  • จัดให้มีมาตรการรักษาความปลอดภัยที่เหมาะสม
  • แจ้งเตือนเมื่อข้อมูลรั่วไหล (Data Breach Notification)
  • จัดเก็บ Audit Log
  • บังคับใช้ Data Retention Policy (ลบเมื่อหมดความจำเป็น)

☁️ Data Processor (ผู้ประมวลผลข้อมูล)

เช่น Cloud Provider,
Payment Gateway

หน้าที่ตามกฎหมาย:

  • ประมวลผลข้อมูล ตามคำสั่ง (Instruction) ของ Controller เท่านั้น
  • ห้ามนำข้อมูลไปใช้งานนอกเหนือจากที่ตกลงในสัญญา (Data Processing Agreement - DPA)
  • ต้องมีมาตรการรักษาความปลอดภัย (Security Measures) ที่ได้มาตรฐาน
  • แจ้ง Controller ทันทีหากพบการรั่วไหลของข้อมูล

Part 3: PDPA vs GDPR

มาตรฐานไทย กับ มาตรฐานยุโรป

🌍 GDPR คืออะไร?

General Data Protection Regulation

กฎหมายคุ้มครองข้อมูลส่วนบุคคลของ สหภาพยุโรป (EU) ซึ่งเป็นต้นแบบที่ทรงอิทธิพลที่สุดในโลก และเป็นต้นแบบของ PDPA ของประเทศไทย

💡 ข้อควรรู้สำหรับ Software Engineer:

หากแอปพลิเคชันของคุณมีผู้ใช้งานเป็นคนในเขต EU (เช่น แอปจองโรงแรมที่นักท่องเที่ยวยุโรปใช้) คุณต้องปฏิบัติตาม GDPR ด้วย แม้เซิร์ฟเวอร์จะอยู่ในไทย!

Part 4: Privacy by Design

การนำแนวคิดไปปรับใช้ในการสร้าง Software

🛠️ Privacy by Design คืออะไร?

"ออกแบบให้มีความเป็นส่วนตัวตั้งแต่เริ่มต้น"

แนวคิดที่ว่า Privacy ไม่ใช่แค่ Feature ที่นำมาแปะเพิ่มตอนเขียนโค้ดเสร็จ (Bolt-on)
แต่ต้องเป็นส่วนหนึ่งของสถาปัตยกรรม (Architecture) ตั้งแต่ขั้นตอนการเก็บ Requirement

กฎหมายบังคับให้ผู้พัฒนาต้องยึดหลักการนี้

📚 7 หลักการของ Privacy by Design (Ann Cavoukian)

1. Proactive not Reactive

ป้องกันล่วงหน้า ไม่ใช่รอให้เกิดเรื่องแล้วค่อยแก้

2. Privacy as the Default Setting

ค่าเริ่มต้นต้อง Private สูงสุด ไม่บังคับให้ผู้ใช้ไปหาปุ่มปิดเอง

3. Privacy Embedded into Design

ฝังอยู่ใน Architecture ไม่ใช่ส่วนเสริม

4. Full Functionality (Positive-Sum)

ไม่ต้องเลือกระหว่าง Security หรือ Privacy ต้องได้ทั้งคู่แบบ Win-Win

5. End-to-End Security

คุ้มครองตลอดวงจรชีวิต (ตั้งแต่เก็บข้อมูล ยันลบทิ้ง)

6. Visibility and Transparency

โปร่งใส ตรวจสอบได้ว่าใครเอาข้อมูลไปทำอะไร

7. Respect for User Privacy

เคารพสิทธิผู้ใช้เป็นศูนย์กลาง (User-Centric)

💻 การประยุกต์ใช้ในฐานะ Software Engineer

Data Minimization

เก็บเฉพาะข้อมูลที่ "จำเป็น" กับฟีเจอร์นั้นจริงๆ (ตอบคำถามสถานการณ์สั่งอาหาร)

Data Masking / Anonymization

ซ่อนข้อมูลใน UI (เช่น เบอร์โทร 081-XXX-XXXX) และทำ Hashing/Encryption ใน Database

Automated Data Deletion

เขียนสคริปต์ลบข้อมูลอัตโนมัติเมื่อหมดอายุหรือหมดความจำเป็น (Data Retention)

🤔 Ethical Dilemma (ปัญหาความขัดแย้งทางจริยธรรม)

ในฐานะวิศวกรซอฟต์แวร์ คุณจะทำอย่างไร?

A. ทำตามคำสั่งเพราะเป็นลูกจ้าง
B. แอบเขียนโค้ดให้ทำงานผิดพลาด
C. อธิบายความเสี่ยงด้านกฎหมาย (PDPA) และเสนอทางเลือกอื่น (เช่น การขอ Permission แบบโปร่งใส)

📜 ACM Code of Ethics (จรรยาบรรณวิชาชีพ)

องค์กร ACM (Association for Computing Machinery) กำหนดจรรยาบรรณที่เกี่ยวข้องกับ Privacy ดังนี้:

  • 1.2 Avoid harm (หลีกเลี่ยงการสร้างความเสียหาย):
    การละเมิดความเป็นส่วนตัวนำไปสู่ความเสียหายทั้งทรัพย์สิน จิตใจ และชื่อเสียงของผู้ใช้
  • 1.6 Respect privacy (เคารพความเป็นส่วนตัว):
    เก็บข้อมูลให้น้อยที่สุด (Minimization) ปกป้องข้อมูลให้ดีที่สุด (Security) และโปร่งใสที่สุด

📝 สรุปท้ายบท (Conclusion)

  • Security ปกป้องระบบ แต่ Privacy ปกป้องสิทธิ์คน
  • PDPA บังคับให้ Developer ต้องออกแบบระบบให้รองรับสิทธิของ Data Subject เสมอ
  • Privacy by Design คือการนำ Privacy ไปฝังใน Architecture ตั้งแต่วันแรก
  • จริยธรรม คือเกราะป้องกันชั้นสุดท้าย เมื่อกฎหมายและเทคโนโลยียังตามไม่ทัน
